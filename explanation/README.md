# Explanation

> **Confidentiality: PUBLIC.** All content in this directory is public platform documentation.

The "why" of Omega -- design rationale, architecture, mechanisms. Read these when you want to understand how a piece of the system works, not how to use it. These are "how does X work" / "why is ..." docs -- if you need "how do I ..." task recipes, see [how-to/](../how-to/README.md); if you need exact signatures, see [reference/](../reference/README.md).

## Information ownership

This bucket owns public platform architecture and design rationale. Internal engine implementation details (CouchDB internals, index structures, replication mechanics) are documented in `omega-knowledge-base`. Project-specific design decisions live in their respective project repos.

### OQL

- [OQL Language](oql-language.md) -- Core language concepts for OQL. Covers terms (the basic unit -- statements that are true or false), symbols and unification (how the engine finds consistent bindings, how unification applies to every term not just `=`), maps and access patterns (`get`/`get-in`/`set` with unification semantics, using `get` as both a reader and a constraint), and links to sub-topic reference pages for datastores, clauses, control flow, built-ins, reducers, and conventions. Keywords: OQL, term, symbol, unification, binding, get, get-in, set, map, constraint, logic programming, Datalog.

- [Execution Model](oql-execution-model.md) -- The most important conceptual doc for writing correct OQL. Covers solution sets (the fundamental table-of-bindings data structure), why terms are not expressions (terms transform solution sets, they cannot be nested inside literals), terms as joins (each term is like a CTE in SQL), clauses as solution boundaries (how in-memory `(:- ...)` clauses keep solutions small, and why `def-query` / `with-query` are deprecated), clause types (stored, in-memory), control flow reference (`when`/`when-not`/`with-table-if`/`and`/`or`/`fold`/`with-count`/`with-order-by`), index matching (queries must match indexes exactly or trigger full table scans), execution semantics (no transactions, fresh context on handler invocation, calling OQL from Clojure via `run-query-clj` returns `[result ctx]` and the `(some? ...)` smoke-test trap, `println-stream` corrupts solution context), and performance rules of thumb. Keywords: solution set, execution model, join, CTE, expression, term, clause boundary, in-memory clause, def-query deprecated, with-query deprecated, index matching, full table scan, no transactions, run-query-clj, result ctx, some? tautology, smoke test, Clojure caller, println-stream, performance.

- [Singleton-row term lookup](singleton-term-lookup.md) -- Why arity-1 terms with an output-only arg (architecturally cardinality-1 rows like `Qo.Schema.Schema/config-data`) have no clean index-match answer today and require an `(enable-full-scan)` bookend. Covers the empty-bound-arg case at index-match time, why neither a defterm nor a `write-index` covers it, the `Qo.Schema.Schema/config-data` worked example (BK-Case3-like marker), adjacent shapes that are NOT this case (by-id lookup, per-parent singleton, cardinality-1 by accident), and what a future engine-support fix would need (empty-pkey defterm or a by-singleton call shape, plus the rekey migration). Keywords: singleton row, arity-1, output-only arg, empty bound args, zero-binding, full-scan bookend, BK-Case3, config-data, schema config, empty pkey, cardinality 1.

- [Agent-step plan shape](agent-step-plan-shape.md) -- Meta-discipline for plans that build a workflow OnEvent (or any large entry-point clause). Covers why plan structure dominates over read docs (a plan that says "add this code to OnEvent's body" produces inline OnEvent regardless of clauses.md), the cargo-cult trap (copying a previous agent step's pre-refactor plan inherits the deep-nesting anti-pattern), the shape every task must produce (one new helper + one `(call ...)` line + Result-envelope update, never inline OQL into existing helpers), the canonical OnEvent target shape (~7-11 lines flat sequential calls, mirroring Strategist `b779ca8`), the orchestrator-review pattern (lighter-weight alternative to the standard `subagent-driven-development` two-stage reviewer subagents — orchestrator reads the diff inline, runs structural / side-effect / spec-compliance checks before marking each task done), and a plan-authoring checklist. Keywords: plan shape, agent-step, OnEvent, ProcessXBatch, flat-sequential helpers, cargo-cult trap, orchestrator review, plan structure dominates, push-run-verify validates behavior not structure, b779ca8, refactor commit, multi-task plan, subagent-driven development, MAB workflow, Strategist, Copywriter, Critic, Analyst, Execute.

Keywords: plugin, install, lifecycle, actor model, event, subscription, continuation, batch processing, Unix philosophy, composable, qo run, debugging, log stream, mango query.
