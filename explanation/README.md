# Explanation

> **Confidentiality: PUBLIC.** All content in this directory is public platform documentation.

The "why" of Omega -- design rationale, architecture, mechanisms. Read these when you want to understand how a piece of the system works, not how to use it. These are "how does X work" / "why is ..." docs -- if you need "how do I ..." task recipes, see [how-to/](../how-to/README.md); if you need exact signatures, see [reference/](../reference/README.md).

## Information ownership

This bucket owns public platform architecture and design rationale. Internal engine implementation details (CouchDB internals, index structures, replication mechanics) are documented in `omega-knowledge-base`. Project-specific design decisions live in their respective project repos.

- [run-term per-row in multi-solution iteration](run-term-per-row-iteration.md) -- When iterating a captured helper once per row in a multi-solution context at scale (K ≥ ~30 rows), use `run-term`, not `call`. Empirically (MAB Allocate V1, K=40 arms): two independent `(call ...)` per-row invocations hung the engine; switching to `(run-term ...)` dropped wall-clock to 3–5s. This is a carve-out from the general "use `call` for in-memory captured helpers" rule — the carve-out applies only to the per-row iteration shape. Covers the updated rule table (call vs run-term by shape), empirical cases from Allocate V1, threshold data (hangs ≥ ~30, proven <30 not tested in this shape), hypothesis on root cause (solution-table residue scaling non-linearly under `call` per-row), and cargo-cult sources. Keywords: run-term, call, per-row, iteration, multi-solution, K rows, captured helper, scale, hang, ECONNRESET, Allocate, BuildAllocationsRow, CountSaForArm, MAB.

- [Code-step pagination pattern](code-step-pagination-pattern.md) -- How a code step (no LLM, deterministic) paginates over a fixed list across multiple Coordinator firings, with revision continuity across batches. Covers the five load-bearing pieces: (1) `ResolveSkip`/`ResolveLimit`/`ResolveOrComputeRevision` reading from `Exec.args[0]` with defaults; (2) deterministic-ordered input read — `prop-vals-by-folder` returns unstable order across invocations; without `with-order-by` on a stable key before the slice, skip-based pagination misses or duplicates ~15-20% of items per drain; (3) `BatchTakeN` slicing `[Skip, Skip+Limit)` with a tail-of-list case; (4) revision-continuity contract — first firing computes `max+1`, subsequent firings inherit via `next-args`; (5) result envelope structure with `next-args: [{step, skip, limit, revision}]` when `has-more=true`, absent when `has-more=false`. Contrasts Execute (filter-based implicit cursor, no positional skip, no revision) with Sync Results (explicit positional skip, revision continuity). Generalises to any future code step that paginates over a fixed list (Allocate, Load Subjects, etc.). Keywords: pagination, code step, Coordinator, skip, limit, revision, batch, drain, next-args, has-more, with-order-by, deterministic sort, prop-vals-by-folder unstable order, BatchTakeN, ResolveSkip, ResolveLimit, ResolveOrComputeRevision, Execute, Sync Results, MAB.

- [LLM output discipline](llm-output-discipline.md) -- Meta-discipline for LLM-driven agent steps when an LLM response can't be parsed by `json-stringify`. Covers the rule (never insert a `string-replace` / regex-clean / preprocessing layer before parsing — fix the producer or file a parser-bug gap), the four objections to scrubbing at consumption (hides interface mismatch, doesn't generalise across edge cases, violates clean-at-boundary architecture, sibling agents prove the prompt is the fix point), what "tighten the prompt" looks like (state the wire format explicitly, show JSON exemplars, constrain the schema, retry-on-parse-failure instead of scrub-and-hope), when to file a parser-bug gap instead of tightening the prompt, and the MAB Copywriter V1 drain worked example where the reflexive `string-replace "\\/" "/"` patch was rejected and the prompt-tightening + engine-gap split was the correct response. Keywords: LLM, agent step, json-stringify, parse failure, string-replace, string-scrubbing, prompt tightening, RFC 8259, control characters, clean at boundary, producer fix, parser-bug gap, MAB Copywriter, Strategist, Researcher, retry on parse failure, data shape ownership.

- [Component-library testing strategy](component-library-testing-strategy.md) -- Strategy + design rationale for unit-testing `qo` component libraries (MAB / SmartLead / GHL / HDC / AI Connector). Covers what unit testing means for OQL (synthetic input + mock at the side-effect boundary + assertion via `(= Expected Actual)`), test-runner design (scratch queries with a convention now, `qo test` later), datastore mocking (per-call `test-<helper>-resp` substitution generalising the proven `test-llm-resp` precedent), agent-step coverage (every named helper + every `with-table-if` branch), and unit vs integration positioning (complement push-run-verify, do not replace it).
