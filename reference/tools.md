# tools/ — Site asset generators

This folder is **gitignored** but kept locally for regenerating SVG assets used by the marketing site at `../site/`.

Every SVG used in the OmegaAI marketing site that has dynamic complexity (animations, computed geometry, dense procedural content) is generated by a Python script here. Hand-editing these SVGs is not recommended — change the script and re-run.

## Why Python?

- Computing path lengths for SMIL stroke-draw animations
- Dense procedural content (network graphs, star fields, Sankey flow conservation)
- Precise mathematical layouts (gauges, ellipses, isometric perspectives)
- Reproducible asset generation across sessions

All scripts use only the Python standard library (`math`, `random`). No external dependencies.

| `gen_sankey.py` | `assets/optimize.svg` | "Optimize" card. Sankey diagram with the L→M/2→R shape (input stack flows to a half-height middle bar then fans back out). Animated. |
| `gen_dashboard.py` | `assets/supervise.svg` | "Supervise" card. HUD-style dashboard with gauge, sparklines, heatmap, progress rings, mini-stats — all draw-on animated. |
| `gen_starfield.py` | `assets/starfield.svg` | The site-wide twinkling background. 220 far + 35 near + 12 bright stars with halos and diffraction spikes. Each star has its own SMIL opacity loop with random phase. |
| `gen_stargate_banner.py` | `assets/omega-stargate-banner.svg` | The hero banner — Ω symbol as a 3D isometric portal. Wireframe. |
| `inline_starfield.py` | Inlines `starfield.svg` directly into `index.html`, `pricing.html`, `consultation.html` | Required because SMIL animations don't run when SVG is loaded via `<img>` or `<object>` over `file://`. The starfield must be inline. |

## Workflow

After editing any `gen_*.py` script:

```bash
cd ~/Development/omega/omega-docs/tools

# Run the script you edited
python3 gen_starfield.py    # or whichever one

# If you regenerated starfield.svg, re-inline it into the HTML files
python3 inline_starfield.py
```

The other SVGs (`connect.svg`, `optimize.svg`, `supervise.svg`, `omega-stargate-banner.svg`) are referenced by the HTML via `<img data-src="...">` and re-fetched on scroll-trigger by `js/main.js`, so just running their gen script is enough.

## Animation pattern (SMIL stroke draw-on)

The cards use a single shared technique: every line/circle/path has a `stroke-dasharray` set to its actual length and a child `<animate>` element that animates `stroke-dashoffset` from `length → 0`. This produces the "drawing itself" effect.

For curves, the dasharray must equal the actual path length (use `2 * pi * r` for circles, `(sweep_deg / 360) * 2 * pi * r` for arcs). A constant dasharray (like `2000`) makes the animation just fade in instead of trace.

## Stagger pattern

Animations across each SVG are staggered by `begin="Xs"`:
- `connect.svg`: edges ripple outward from center (distance-based begin times) over 0.4-2.5s
- `optimize.svg`: grid → corner brackets → flow bands left→right → middle bar → ticks (0-2.5s)
- `supervise.svg`: panels first, then widget interiors per-tile (0-3s)
- `starfield.svg`: each star has a random `begin="-Xs"` (negative = pre-shifted phase) so they're all out of sync from frame 0

## Re-triggering animations on scroll

`js/main.js` has an IntersectionObserver that swaps `data-src` → `src` (with a cache-buster) when a card enters the viewport. This re-fetches the SVG, which restarts SMIL animations from t=0. The cards animate exactly once per page visit (the observer unobserves after first trigger).

## Site brand colors

- Magenta primary: `#e84bfa`
- Deep purple secondary: `#351c75`
- Background base: `#121212` (Material UI dark)
- Card background: rgba(18, 18, 22, 0.85) with backdrop blur

All scripts hardcode these. Change them in the script and regenerate.

## Why this folder is gitignored

These scripts are auxiliary build tooling — not part of the deployed site. The committed `.svg` files in `../site/assets/` are the source of truth for the website. This folder exists so future sessions can regenerate or modify the assets, but it doesn't need to be in version control.
