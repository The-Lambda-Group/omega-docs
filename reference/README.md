# Reference

> **Confidentiality: PUBLIC.** All content in this directory is public platform documentation.

Factual lookup for Omega APIs, schemas, and IDs. Read these when you need exact signatures, protocol definitions, or command inventories. These are "what is ..." docs -- if you need "how do I ..." task recipes, see [how-to/](../how-to/README.md); if you need "why does ..." understanding, see [explanation/](../explanation/README.md).

## Information ownership

## Navigation map

### Language reference

- [OQL hard rules](oql-hard-rules.md) -- Non-negotiable rules an agent must never violate when writing OQL. Covers forbidden datastore opens (`system/`, `omega/query-omega/data/...`, `Qo.Data.*/`), forbidden operations in scratch files (any `!`-suffixed write, `delete!`, `refresh`), the absolute prohibition on `refresh` (breaks CouchDB indexes), and the string-vs-boolean requirement (`"true"`/`"false"` only -- native `true`/`false` produces silent wrong answers). - [Clauses](clauses.md) -- OQL clause syntax and semantics. Opens with **clause size and decomposition discipline** (5–10 lines per body, no nested `with-table-if`s, compose via `(call SmallHelper ...)` not inline `(and ...)` blocks, every helper declares its own `{"capture" [...]}`) — read this section first when starting any workflow OnEvent or large entry-point clause. Then covers stored clauses (the `!` read/write convention with datastore namespaces), in-memory clauses (local function-like definitions), inner clauses (experimental, with known limitations on stored writes), and closures/capture (threading values across clause boundaries with the `{"capture" [...]}` syntax). Includes a dedicated subsection **"Captures are not transitive"** — captures do NOT propagate through `call`; each clause that invokes `(call X)` must declare `X` in its own `{"capture" [X]}`; building a two-level chain (OnEvent → ProcessAnalyst → {ThinkPhase}) fails at runtime with `TermExecutionRunnable found for class: clojure.lang.Symbol`; fix is to flatten and have the entry-point OnEvent capture every helper it will ever call directly. Keywords: clause, `:-`, stored clause, `!` suffix, in-memory clause, inner clause, capture, closure, functor, clause size, decomposition, helper, BuildCtx, ResolveCtx, thin OnEvent, flat sequential, project lore, design discipline, debugging discipline, stack trace breadcrumb, captures not transitive, nested capture fails, TermExecutionRunnable, clojure.lang.Symbol, flatten captures, two-level call chain.

- [Built-in terms](built-ins.md) -- The complete built-in term inventory. Opens with a negative-space callout listing primitives that look like they should exist but don't (`string-length`, `string-take`/`substring`, `keys`, `assoc` as a fold reducer) with canonical workarounds — check there first when a name "feels obvious." Covers core terms (`=`, `defined`, `return`, `uuid`, `time-now`, `throw`), arithmetic (`+`, `-`, `*`, `/`, `>`, `<`, `>=`), random (`rand` — uniformly random double in `[0.0, 1.0)`, non-cryptographic, non-seedable; `rand-int Min Max Output` — uniformly random integer in caller-specified `[Min, Max)`, throws `RAND_INT_UNBOUND_ARG`/`RAND_INT_NON_INTEGER`/`RAND_INT_BAD_RANGE` on bad input), map operations (`get`, `get-in`, `set`, `merge`, `select-keys`, `zip`), array operations (`get-list`, `length`, `append`, `concat`, `partition`), string operations (`string-split`, `string-concat`, `string-contains`, `string-replace`, `re-find`, `json-stringify` including the bidirectional parse/stringify contract and the parse-probe pattern where wildcard `_` fails but a named symbol succeeds), data transformation patterns (preparing data for `write-table`), the symbols-in-literal-arguments gotcha (why you must bind maps or key vectors with `=` before passing them to terms, including `row-index` key lists and `throw` data maps), and the HTTP term (`http-request` with request/response map keys, plus deprecated `http-get-request`/`http-post-request`). The `throw` section documents the two-arg signature `(throw "TAG" DataMap)` and explains why `(throw "TAG" {map-literal})` fails at runtime — the map literal is evaluated at parse time before symbols are bound, so `DataMap` must always be a pre-bound variable. Keywords: built-in, term, unify, get, set, merge, zip, get-list, json-stringify, json-parse, parse probe, wildcard, string-concat, http-request, http-get-request, symbols in literals, row-index, key vector, string-length, string-take, substring, keys, assoc, nonexistent primitives, missing primitives, negative space, what doesn't exist, rand, rand-int, random, random number, double, integer, range, sampling, probability, RAND_INT_UNBOUND_ARG, RAND_INT_NON_INTEGER, RAND_INT_BAD_RANGE, throw, throw signature, throw map literal, throw runtime error, typed exception, error tag, error data map, parse time evaluation.

- [Control flow](control-flow.md) -- The most critical reference doc. Covers universal quantifiers (`when`, `when-not`, `if` -- these are "if any" operators, not per-row), per-solution branching (`with-table-if` -- condition evaluation, schema/binding-list as JOIN key, branch scoping), the existence-check fallacy (`with-table-if` runs the then-branch ONCE PER CONDITION SOLUTION, not once -- a multi-row condition like `prop-vals-by-folder` with K rows runs the then-branch K times, producing K output solutions; to get existence-check semantics fold the condition to a single solution first), preserving per-row identity (the `SELECT DISTINCT` collapse problem and why schemas need unique-per-row columns), dropping rows from solutions (`(= true false)` idiom), throw-in-else-branch does NOT halt the enclosing clause (variables unbound in the else-branch remain as `clojure.lang.Symbol` objects; terms after the `with-table-if` still execute -- move dependent terms inside the then-branch), batch-wide vs per-row conditions (`defined` is lexical, not per-row), nesting costs and the flat-sequential alternative, the check-then-rebind idiom, the Ctx-threading pattern, binding-list association rules, and the absence of a `not` operator. - [Datastores](datastores.md) -- Datastore declarations, functors, defterm, and write-index. Covers how `(datastore Name "path")` binds a symbol to a CouchDB location, how `(functor "name" arity Ds Func)` creates callable references, how `(defterm Ds {...})` declares term schemas with primary keys to create indexes, and how `(write-index Func "path" ["args.N"])` creates secondary indexes. Keywords: datastore, functor, defterm, write-index, primary key, secondary index, CouchDB, term schema, arity.

- [defterm — semantics and migration consequences](defterm.md) -- The presence-vs-absence rules `datastores.md` does not cover. `defterm` is a relatively new construct; many terms have no defterm and that is the normal default. A term **without** a defterm uses CouchDB-assigned (random) row ids; a term **with** a defterm uses content-addressed ids derived from `id_<sha256(pkey)>`. Therefore **adding a defterm to a term that already has data is a destructive rekey** — existing rows stay at their old ids and become unreachable through pkey lookups (the engine computes the new deterministic id and finds nothing). This is NOT a fix for `omega/query/full-scan` on a no-defterm term — the right fixes are caller-restructure (default), `(write-index …)` (no rekey, additive), or a bookend with explicit user authorization. Adding a defterm is a separate data-migration project. Keywords: defterm, primary key, row id, deterministic id, sha256, pkey hash, content addressing, CouchDB-assigned id, random UUID, migration, rekey, destructive, data loss, full-scan triage, write-index, bookend, when to add, when not to add, absence is normal.

- [Reducers](reducers.md) -- Solution table aggregation operations. Covers `fold` (solution-wide reduce with init/sym/func/agg), `with-unique-by` (changing what "unique" means for fold -- essential when fold symbols have few distinct values), `with-order-by` (always ascending — use `prepend` instead of `append` to produce a descending list), `with-group-by`, `with-count`, `with-limit` (not yet implemented -- will crash). Includes critical pitfall documentation: fold is solution-wide not per-row, fold folds over unique values not rows, why `batch=1` masks fold bugs, and **fold over zero solutions does not bind its accumulator** (the `Init` arg is the reduction seed, not an empty-case default — empty upstream produces an unbound `Agg` and a downstream `omega/query/no-return`). Documents the canonical fold-with-default idiom: `with-count` + `with-table-if (> Cnt 0)` to gate the fold and bind the seed in the else branch, plus the `with-count` + compound-condition NPE workaround (wrap the compound predicate in an in-memory clause). Also links to the partition-scaling performance explanation: `fold` / `with-group-by` / `with-order-by` scale non-linearly with partition count and the idiomatic fix is to wrap the reducer body and invoke it via `run-term` per partition. Keywords: fold, reduce, aggregate, with-unique-by, with-order-by, with-group-by, with-count, with-limit, append, prepend, descending, sort direction, ascending only, per-row vs solution-wide, partition, run-term, performance, zero solutions, empty input, fold default, no-return, fold-with-default idiom, NullPointerException.

- [Query execution](query-execution.md) -- The `with-skip`/`with-limit` constraint for CouchDB view pagination. Covers the critical limitation that skip/limit only work when the runtime resolves through a view index -- when it falls back to Mango scan, skip/limit are silently ignored and the query returns up to 1,000,000 rows. Keywords: with-skip, with-limit, pagination, view index, Mango scan, silent bug, page-query, CouchDB.

- [Prefix-scan index selection](prefix-scan-index-selection.md) -- How `query-term-from-table` selects between defterm PK lookup, exact write-index bind, prefix-scan (single row), prefix-scan with k-way heap merge (N rows), Mango fallback, and no-matching-index error. Includes the operator push-down table showing which strategies honour `with-limit`, `with-skip`, and `with-sort`, the prefix-test algorithm, endkey sentinel formula, and heap-merge primitive descriptions. Keywords: index selection, prefix scan, write-index, defterm, PK lookup, Mango fallback, with-skip push-down, with-limit push-down, with-sort push-down, k-way merge, heap merge, lazy-range-fetch, startkey, endkey, no-matching-index, omega/query/no-matching-index, operator push-down, strategy selection.
