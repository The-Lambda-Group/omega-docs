Singleton-Row Term Lookup

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

Some terms hold exactly one row by design — a global config, a singleton service descriptor, a per-cluster invariant. The natural call shape for reading that row is arity-1 with the single arg as the output:

(Qo.Schema.Schema/config-data Config)

There is no input arg to constrain on, no parent id to look up by, no key to bind. This shape sits at the seam of the engine's index-matching model and currently has no clean answer in OQL.

What the engine does at index-match time

The OQL engine picks an index by inspecting which arg positions are bound at the call site, then matching that bound-arg set against the term's primary-key positions and any declared write-index field sets.

For an arity-1 term where the only arg is the output:

  • Bound-arg set: empty ({}).
  • pkey field set on the defterm (if any): a non-empty list of arg names — pkey requires bound positions to compute the deterministic id.
  • write-index field sets: each declares one or more arg positions; an empty bound-arg set matches none of them.

The empty bound-arg set cannot construct any view-key prefix, so no view query is possible — Mango is the only fallback. That fallback fires the omega/query/full-scan throw.

Why a defterm doesn't help

A defterm with "primary-key" [] is not a documented or supported declaration — it would require engine support to recognize the empty pkey as a known sentinel and route lookups through a single deterministic id. That work has not been done.

A write-index also does not help: a write-index declares which arg positions to materialize a view over. With no bound args, every view's start-key is empty and the runtime cannot pick one over another.

The working answer today

The only way to read the row through the term today is to bookend the call with (enable-full-scan) / (disable-full-scan). The bookend stays as a placeholder for a missing engine pattern, not as an over-binding fix that was deferred. Mark it in source with a comment pointing back to this doc so it can be retired when the engine pattern arrives.

(:- (Qo.Schema.Schema/create-config! Config)
    (enable-full-scan)
    (Qo.Schema.Schema/config-data Config)
    (disable-full-scan))

What "singleton" does and doesn't mean

The singleton case requires architectural cardinality 1 — the term schema enforces one row by construction, and the read has no input arg available because there is no input dimension to bind.

Do not reach for "singleton lookup" framing when:

  • By-id lookup of a known row — if the caller already has the row's id, pkey, or any indexed field, bind that field and the term reads through the existing index.
  • Per-parent singleton — a term that holds one row per parent (e.g., one config row per AppId) is a normal pkey-on-parent-id lookup — bind the parent id, hit the pkey index.
  • Cardinality-1 by accident — a term that happens to have one row in a particular environment but is not architecturally singular has a normal index.

What a future fix would look like

Resolving the singleton case requires engine support — none of this is implemented today:

  1. Empty-pkey defterm support — allow "primary-key" [] in a defterm declaration. The deterministic id becomes a single fixed digest. Would also need the bulk-insert path to recognize the empty pkey case.
  2. A by-singleton term call shape — a built-in like (singleton-of <Term> Result) that bypasses the index-match step entirely and reads the cardinality-1 row by a known reserved id.
  3. Migration of existing data — any term that already has a singleton-shaped row written through the legacy path has a CouchDB-assigned random id; switching to either approach above would orphan the existing row exactly the same way that adding a defterm to a populated term does.

Until one of these lands, the bookend is the working answer.
