defterm Semantics

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

defterm declares a term schema with a primary key on a datastore. It is a relatively new construct in OQL — it was never a hard requirement, and many existing terms in the system have no defterm, which is the normal default state.

Read this before you "add a defterm to fix a full-scan throw." Adding a defterm to a term that already has rows is a destructive rekey, not an additive index.

Syntax recap

(datastore Qo.Data.LogStream "omega/query-omega/data/log-stream")

(defterm Qo.Data.LogStream
  {"name" "log-stream-data"
   "args" ["AppId" "LogStreamId" "LogStream"]
   "primary-key" ["LogStreamId"]})

Names the term, lists positional args, declares which arg(s) are the primary key.

What a defterm controls: row-id derivation

The single most important fact about defterm is that it changes how each row's _id is generated when the term is written.

Without a defterm (the default)

A term without a defterm is written through the legacy bulk-insert path. The row goes to CouchDB's _bulk_docs endpoint with no _id field, and CouchDB assigns the row id. The row id is opaque — the engine reads the row back through Mango selectors or view indexes, never by id.

With a defterm

A term with a defterm is written through the termdef path. The row id is derived from the primary-key value:

_id = "id_" + sha256(json.write_str(pkey-args))

The same pkey value always produces the same _id. This is why defterm-backed writes are upserts, why fetch-revs is needed before a bulk write (to find the existing rev), and why deleted defterm rows can be cleanly resurrected with a new write.

Why this matters at the language level

Three downstream consequences follow from the row-id rule:

  1. Defterm pkeys behave like SQL primary keys. Two writes with the same pkey value collide on the same row id; the second supersedes the first. Without a defterm, repeated writes append new rows.
  2. Defterm-backed terms can be looked up by exact pkey at view-index speed. The pkey is the row id, so lookups by full pkey are O(1) hash-style fetches.
  3. Stack traces show positional pkey matches against the defterm. When triaging an omega/query/full-scan throw, the engine checks bound-arg positions against the defterm's pkey arg positions.

When to add a defterm

Add a defterm at the time you create a new term if:

  • You want pkey-based upsert semantics (writing the same pkey overwrites instead of appending).
  • You want the by-pkey lookup to be a cheap deterministic fetch.
  • The pkey is well-defined and stable across the term's lifetime.

If you are not sure, leave the defterm off. A term without a defterm is fully functional — the absence is normal, not a defect. You can always add (write-index …) declarations later for off-pkey lookups; that path does not rekey existing data.

When NOT to add a defterm

  • Do not add a defterm to a term that already has data as a way of "fixing" a query problem.
  • Do not read the absence of a defterm as evidence of an orphan, a missing source artifact, or a bug.
  • Do not add a defterm "just to make it more complete." Defterms exist to enforce a uniqueness/upsert contract. If the term has no real uniqueness contract, declaring one introduces new failure modes without a corresponding gain.

Adding a defterm to a term with data — this is destructive

If a term has been in production with no defterm and accumulated rows, those rows have CouchDB-assigned (random-UUID) row ids. The engine has been reading them through Mango selectors, view indexes, or the legacy term-query path.

If you then add a defterm and deploy:

  1. The deploy registers the new defterm.
  2. Subsequent writes go through the termdef path: row id = id_<sha256(pkey)>.
  3. Subsequent reads by pkey compute the deterministic id and try to fetch it directly.

The pre-existing rows still have their old random-UUID ids. The deterministic id the engine computes from the pkey does not match any existing row. From the perspective of a pkey lookup, the existing data is unreachable — even though the rows are still physically present in CouchDB. From the application layer, this is indistinguishable from data loss.

This is NOT something (refresh) fixes. It is NOT something a redeploy fixes. It is NOT something the engine reconciles in the background. It is a content-addressing change with no automatic migration.

What an actual defterm migration would require

  1. Read every existing row of the term into memory (or stream it).
  2. Compute the new deterministic id for each row from the pkey.
  3. Write each row at the new id (the termdef-path write does this if you re-insert through the term).
  4. Delete the old row at the old id.

This is a separate data-migration project, not a side effect of declaring the defterm. If you find yourself reaching for "let me just add a defterm to this term," and the term already has production data, stop and ask the user.

Removing a defterm from a term with data

The mirror case. Removing a defterm from a term that has been written with deterministic ids causes subsequent writes to go back through the legacy path with random ids. The existing deterministic-id rows become orphans relative to the new write path. The same migration discipline applies.

Practical implication: full-scan triage

When a query throws omega/query/full-scan and the offending term has no defterm, an inexperienced agent might conclude "the term has no pkey index, so I should add a defterm." This is the wrong move on a term with data.

The valid fixes for a full-scan throw on a no-defterm term, in priority order:

  1. Caller-restructure — the default. Drop over-bound args from the call; bind only what an existing index covers.
  2. Add a (functor …) + (write-index …) on the existing datastore. This adds a secondary index without rekeying any existing rows.
  3. Bookend with (enable-full-scan) / (disable-full-scan) at the narrowest scope, with explicit user authorization, as an interim.

Adding a defterm is not on this list. If a defterm is genuinely the right contract for the term going forward, it is a separate project that includes the data-migration step — not a one-line declaration deploy.
