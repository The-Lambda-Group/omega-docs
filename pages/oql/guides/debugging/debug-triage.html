Debug & Triage

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

This is the triage mirror of develop-oql. Authoring runs the push-run loop forward — add one step, verify, add the next. Triage runs the same loop backward — strip one step, verify, strip again. Both depend on the same tight feedback cycle. If you are debugging an OQL failure, you are the authoring loop in reverse — read the develop-oql guide first if you haven't, then come back.

Every rule below produces an artifact. A minimum-repro command, a ruled-in/out list, a written Observed/Hypothesis/Test trio. If you finish triage without writing anything concrete down, you ran on vibes and you are going to ship a non-fix.

The six triage rules

1. Minimum reproduction is the first move, always

Before any theory of cause, strip the failing invocation to the smallest thing that still reproduces. If what you are running is larger than what reproduces, you have not narrowed.

Artifact: a one-screen scratch file or qo run command that reproduces the failure. Pin its exact output at the top of your working notes.

2. Shrink the frame before changing any code

Comment out terms from the bottom of the clause body up, moving the (return [Result]) earlier at each step. At each position, ask: does the failure persist? When the failure disappears, the removed term (or its interaction with something above) is the cause.

3. Redirect Result to the failing term's inputs

The failing term tells you that it failed. Its inputs tell you what data it failed on. Before any theory, bind Result to the inputs of the failing term and run. The bug is almost always visible in the data.

;; Before: the failing term
(some-failing-term Arg1 Arg2 Output)
(return [Output])

;; After: inspect the inputs
(= Result [Arg1 Arg2])
(return [Result])

4. Keep a ruled-in / ruled-out list

For each probe, record: what you ran, what came back, and what it rules in or out. This is not optional — without the list, you loop. With it, you build toward the root cause.

5. Never pattern-match a prior fix

If this error looks like an error you've seen before, resist the temptation to apply the same fix. Confirm the specific failure mode first. Pattern-matching prior fixes without confirmation is how wrong fixes get shipped.

6. Stop and surface when you're stuck

If you have applied a fix and the error persists, or if you have applied two different fixes and neither worked, stop. Describe the failure in terms of what you observed (not what you theorized) and surface to the user. Do not retry a third fix without input.

Triaging omega/query/full-scan

When a query throws omega/query/full-scan:

  1. First move: caller-restructure. Drop over-bound args from the call; bind only what an existing index covers. This is the fix in the majority of cases. An over-bound call is one that binds more args than any declared index covers, forcing a Mango fallback and triggering the throw.
  2. Check the bound-arg set against the defterm pkey and write-indexes. The error message includes :bound-positions (which args are bound) and :candidate-indexes (which indexes were considered). Compare directly.
  3. If no index serves the query and caller-restructure is not possible: add a (functor …) + (write-index …) on the existing datastore. This adds a secondary index without rekeying any existing rows. Requires user permission.
  4. As a last resort: bookend with (enable-full-scan) / (disable-full-scan) at the narrowest scope, with explicit user authorization, as an interim while the proper fix is scheduled.
  5. Never add a defterm to fix a full-scan on a term with existing data. This is a destructive rekey — see the defterm doc.

If the offending term is a singleton-row read (arity-1, output-only arg), see the Singleton-Row Lookup doc — caller-restructure does not apply there because there are no input args to restructure.

Triaging omega/query/no-return

omega/query/no-return is a single error name that covers three structurally different failure modes:

ModeTriggerFirst check
Scratch-file no-returnA scratch query binds Result but never calls (return [Result])Add (return [...]) — see Scratch Queries guide
Dispatch-layer no-returnThe dispatcher could not select a clause to runCheck arity (must be 2: Exec, Result), check Clauses map, check protocol spec
Body-level zero-solutionsThe clause body ran but a term produced zero solutionsBisect the body — return early at each term to find where the solution table empties

Dispatch-layer diagnosis

If the no-return fires before the clause body runs, the dispatcher failed to select a clause. Check:

  • Arity — the implementation clause signature must be exactly (:- (MyClause Exec Result) ...). A 3-arg signature never unifies.
  • Clauses map — the set-implementation-clauses call must map the protocol method name (e.g., "run", "on-event", "install") to the correct clause symbol.
  • Protocol spec — the protocol ID and method name must match exactly.

Body-level diagnosis

Bisect the body by returning early at each term. When you find the term where the solution table empties:

  • Unbound fold accumulator: the most common cause. A fold over zero upstream solutions leaves the accumulator unbound. Use the canonical fold-with-default idiom (see Reducers).
  • Missing field in a get/get-in: the term fails per-row for rows where the field doesn't exist. Check the key name and the data shape.
  • Index miss: a datastore lookup found no matching rows. Check the bound args against the index.
  • LLM response truncation: a call-llm invocation fails with omega/jvm/EOFException: JSON error (end-of-file inside string). The LLM emitted a response that was cut off mid-string before the closing braces and quotes that would make the payload valid JSON. This looks like a JSON parse bug but is not. Fix the producer — tighten the LLM prompt so it emits complete JSON responses within token limits.

Check data table integrity

Use this when a table-backed lookup fails, a row reads as empty, or a helper probe produces surprising row-page results. Before diagnosing helper logic, prove whether the property table's storage contract is intact.

The common invariant is:

logical row key -> row-index entry -> row page -> required row content

If any link is missing, helper probes that depend on the same table are not clean evidence about the helper.

When to run this check:

  • A helper that has worked before suddenly returns no rows.
  • A write confirmed success but a subsequent read finds nothing.
  • A row-index or sec-index lookup returns unexpected results.
  • Any time you are about to blame the helper logic but haven't yet verified the table has the row you expect.

The check:

  1. Probe the row-index directly: (Qo.Public.OqlApi.Db.Prop/row-index FolderId RowKey PageId). Does it return the expected PageId?
  2. Probe the page directly: (Qo.Public.OqlApi.Page/page-by-id PageId Page). Does the page exist?
  3. Probe the row content: (Qo.Public.OqlApi.Db.Prop/prop-vals-by-folder FolderId PageId PropVals). Does the page have the expected property values?
  4. If any link is missing, the data integrity is broken — not the helper logic.

Triage in the worktree, not in scratch

Scratch queries are not always a clean triage environment. When the behavior you are investigating depends on multi-solution context — for example, a step that runs across many arms or rows simultaneously — a standalone scratch query cannot reproduce that context. Timings will be misleading, and a fix validated in scratch may not hold in the real component.

When scratch gives misleading results, switch to worktree triage. The approach mirrors the frame-shrink discipline in rule 2, but applied to the live running implementation:

  1. Open the implementation file in the affected project's worktree. Do not create a parallel scratch file.
  2. Strip terms from the bottom of the clause body up. Comment out terms one at a time, exactly as you would move the (return [Result]) earlier in scratch. At each position, push the change and run against the real component.
  3. Observe the result with each change. You are narrowing the frame of the live component, not the frame of a simulation. This produces timings and error messages that match production.
  4. Restore the terms once you have identified the cause. Worktree triage is a read-probe cycle, not a rewrite. Restore bottom-up exactly as you stripped top-down.

Worktree triage is the correct first move whenever the failure involves scaling, multi-solution context, or cross-component state that a scratch query cannot replicate.

println-stream corrupts solution context

println-stream has a side effect: it modifies the solution context. If you call println-stream before binding your Result symbol, the solution context may be corrupted and Result becomes unresolvable. Always bind Result first, then log. Prefer the print-stream query (Omega Queries component) over raw println-stream.
