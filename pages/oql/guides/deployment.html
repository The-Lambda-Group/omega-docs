Library-side vs installed-side

Every component exists in two places in the workspace:

  • Library-side (dev) component — the component in your developer library. qo push [impl-file] updates this one. The impl file's embedded set-implementation-clauses script calls the engine with $$ARG$$ as the impl-id; at push time $$ARG$$ resolves to the library-side impl-id.
  • Installed component — a separate copy of the component installed into a specific workspace location (e.g. a MAB Agent System page). It has its own component-id and its own impl-id. qo push does not touch the installed component.

If you push a fix and the running workflow continues to exhibit the old behavior, you have fixed the library side only. The installed component is still running the old clauses.

Deploying a fix to a running installed component

  1. Push the updated impl file: qo push [impl-file]. This updates the library-side clauses.
  2. Identify the installed component's impl page (e.g., "Component Installs/My App/Implementation").
  3. Run the targeted deploy against the installed component's impl page to update it with the new clauses. The exact command depends on the deployment protocol your component uses.

The two steps are always separate. Never assume a library-side push has updated a running installed component.

Test at realistic batch sizes

Always test at realistic batch sizes before declaring working. If your pipeline runs at 100 rows per batch, test at 100 — not at 1.

Why this matters — several OQL constructs have behavior that depends on row count. A query that works at limit=1 can fail at limit=100:

  • with-table-if schema collapse only bites when rows share schema values. At limit=1 the collapse is invisible.
  • fold cross-row merge — a fold that returns the correct single row at batch=1 returns a merged all-rows array at batch>1.
  • Cardinality multipliers from captured helpers are invisible at batch=1 but produce exponential blowup at scale.
  • call vs run-term in per-row iteration — call hangs at K~40 rows in a multi-solution iteration; the hang only surfaces when you test at realistic scale.

The discipline: run at least one batch at the expected production batch size before merging. The "it works at limit=1" report is not a signal of correctness — it's a signal that you haven't tested the constructs that only fail at scale.

Verify Thompson-sampling allocation steps

A cargo-cult-able verification pattern for any OQL code step that runs Thompson sampling to distribute a budget across arms. Covers four invariants: variance is real (sampling is not deterministic), no arm starvation, budget tracking is correct, and selection pressure flows from prior strength.

The pattern: N-firing drop-between distribution sanity

Run the step multiple times against a clean slate, record per-arm counts, then check the four invariants:

  1. Variance is real. Run the step N times (e.g., 10). If every run produces the same per-arm allocation, sampling is deterministic — something is wrong. The allocations should vary across runs.
  2. No arm starvation. Over N runs, every arm should receive at least one allocation. An arm that never receives an allocation despite having a non-zero prior suggests the prior is being computed incorrectly or the arm is being dropped before sampling.
  3. Budget tracking is correct. The sum of all per-arm allocations across all runs should equal N × total-budget. No budget should be lost or double-counted.
  4. Selection pressure flows from prior strength. Arms with stronger priors (higher alpha, lower beta) should receive proportionally more allocations over N runs. Compute the per-arm fraction and verify it correlates with prior strength.

If any invariant fails, the allocation logic is incorrect. Use debug-by-returning-early to inspect the intermediate state at each step.

Calling OQL from Clojure

When you invoke OQL from Clojure (rather than via the CLI, MCP, or HTTP API), the entry point is omega-db.run/run-query-clj. It returns a two-element vector [result ctx]:

  • result — whatever the last term produced.
  • ctx — the query context after evaluation, including the solution table and accumulated :failed-at frames.

Gotcha: (some? (run/run-query-clj q)) is always true. The return value is a vector, and a non-empty vector is non-nil — even when the query produced zero solutions, threw a caught error, or wrote nothing. Always destructure and assert on what the query is supposed to produce:

;; WRONG — this passes even when the query is a no-op or fails internally
(is (some? (run/run-query-clj '((MyDs/init)))))

;; BEST — destructure and assert on the query's expected output
(let [[result _ctx] (run/run-query-clj '((MyDs/get-thing "id" Thing) (return [Thing])))]
  (is (= expected-thing result)))
