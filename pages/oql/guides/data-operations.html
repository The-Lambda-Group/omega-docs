Drop rows with forced unification failure

How to deliberately exclude rows from a solution table in OQL using the (= true false) idiom.

When to use this

Any time you want a with-table-if branch to drop rows rather than write them. Typical use cases:

  • Filtering out HTTP responses that shouldn't be processed (e.g. 422s)
  • Excluding rows that match a sentinel value from an otherwise-writable solution
  • Short-circuiting a branch during debugging

The idiom

;; Drop rows where HttpStatus is 422
(with-table-if (= HttpStatus 422)
  [AgentLink HttpStatus Result]
  (= true false)    ;; then: fail the branch — row drops from the solution
  (= true true))    ;; else: pass through unchanged

(= true false) attempts to unify two values that cannot match, which fails the branch. In the SQL UNION analogy from Control Flow, this is equivalent to SELECT ... WHERE 1=0 for one of the two CTEs — the rows matching the condition simply don't appear in the output.

Handle HTTP 422 responses in status dispatch

A dispatch that explicitly handles 2xx and 400 doesn't match 422. A with-table-if branch that matches only (or (<= 200 HttpStatus 299) (= 400 HttpStatus)) leaves 422 and every other non-200/400 status unmatched. Those rows fall through to the downstream table with their branch-scoped symbols unbound. write-table then tries to materialize those rows, can't resolve the unbound symbols, and errors with WRITING_SYMBOLS.

The fix: add an explicit filter before the write gate, using the row-drop idiom:

;; Drop 422s before the write gate
(with-table-if (= HttpStatus 422)
  [AgentLink HttpStatus Result]
  (= true false)    ;; drop 422 rows
  (= true true))    ;; pass everything else

;; Now safe to dispatch on 2xx vs 400
(with-table-if (<= 200 HttpStatus 299)
  [AgentLink HttpStatus Result]
  (= Result {"status" "success"})
  (= Result {"status" "error"}))

Code-step pagination pattern

A code step is a workflow agent step with no LLM call — it reads input data deterministically, calls an adapter or performs a computation, and writes results. When the input list is too large to process in one Coordinator firing, the step must paginate: fire once per batch, pass a cursor forward, and stop when the list is exhausted.

The five load-bearing pieces

1. Skip / Limit / Revision input args

OnEvent reads three optional args from Exec.args[0]:

  • skip (default "0") — positional cursor into the sorted input list.
  • limit (default "1") — how many items to process this firing. Caller controls batch size; the implementation never hard-codes it.
  • revision (optional) — if present, this firing is continuing an in-progress drain at the named revision; if absent, compute a fresh revision.

All three are numeric strings (OQL convention). Parse with json-stringify before arithmetic.

2. Deterministic sort

The input list must be sorted deterministically across firings. If the sort order changes between firings, skip/limit will return wrong items on the second firing. Use a stable sort key (UUID, created-at timestamp, or any value that doesn't change between firings).

3. Revision continuity

A revision arg binds all firings in a multi-batch drain to the same logical snapshot. The first firing computes a fresh revision (e.g., a UUID). Subsequent firings pass the same revision back in. The step reads data at the revision snapshot, not the current live state — this prevents a mid-drain write from changing the count or order of items and causing the drain to skip or double-process items.

4. has-more signal

The result envelope must include a has-more field. When the step processes its batch and finds that skip + batch-size < total-count, it returns "has-more": "true" and the next-args needed for the Coordinator to fire again. When skip + batch-size >= total-count, it returns "has-more": "false" and the drain is complete.

5. Compute has-more and next-args

(:- (ComputeExecHasMore Skip Limit TotalCount HasMore NextSkipStr)
    (+ Skip Limit NextSkip)
    (with-table-if (< NextSkip TotalCount)
      [Skip Limit TotalCount HasMore NextSkipStr NextSkip]
      (and (= HasMore "true")
           (json-stringify NextSkip NextSkipStr))
      (and (= HasMore "false")
           (= NextSkipStr "0"))))

Canonical exemplars

  • Execute step: introduced ResolveLimit, BatchTakeN, and ComputeExecHasMore.
  • Sync Results step: cargo-culted those three helpers, added ResolveSkip and ResolveOrComputeRevision, and established the deterministic-sort requirement.
