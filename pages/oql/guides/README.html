OQL Guides

Step-by-step recipes for working with OQL. Each doc answers a specific "how do I..." question. Read the develop-oql guide first — it is the foundation everything else builds on, and every OQL implementation plan must list it as the single most important doc.

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

Contents

  • Develop OQL — The REPL-driven authoring and triage manual. Operating rules, the inspect-by-mutating-Result mechanic, the push-run loop, and both the forward (authoring) and backward (debugging) directions of the same loop.
  • Debug & Triage — Minimum reproduction discipline, narrowing an OQL failure by stripping the frame, triaging full-scan and no-return errors, returning early to inspect intermediate state, and the check-data-table-integrity procedure.
  • Scratch Queries — How to write scratch queries that always end with (return [Result]), what scratch files can and cannot do (read-only), and the count-rows-in-a-defterm CouchDB doc_count recipe.
  • Deployment — Library-side vs installed-side components, deploying a fix to a running installed component, testing at realistic batch sizes, and the Thompson-sampling allocation step verification pattern.
  • Data Operations — Drop rows with forced unification failure, handle HTTP 422 responses in status dispatch, and the code-step pagination pattern for multi-batch coordinator firings.
  • OQL Queries vs Component Implementations — The critical distinction between .oql library files (deployed via omega-cli run-query) and JS component implementations (deployed via qo push), and when to use omega-cli vs qo.
