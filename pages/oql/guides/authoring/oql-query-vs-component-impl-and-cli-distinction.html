OQL Queries vs Component Implementations, and the Two CLIs

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

The recurring confusion

Agents and developers working in the Omega ecosystem frequently make two related mistakes:

  1. Treating any .oql file as a "component implementation" and trying to deploy it with qo push.
  2. Using qo when the task requires omega-cli, or vice versa.

This doc draws the line clearly. Read it before writing any plan or task that involves deploying OQL code.

Two kinds of OQL artifact

OQL queries — scratch queries and library clauses (.oql files)

An OQL query is a .oql file containing OQL clause definitions (:- heads and bodies). The canonical examples are:

  • Scratch queries — one-off queries in oql/scratch/, gitignored, used for probing, backfill, or migration. They end with (return [Result]).
  • Library queries — reusable clause definitions in oql/queries/ (e.g. oql/queries/add-functors/00-tbl.oql). They define named clauses that other queries can call.

How they are deployed: via omega-cli run-query <file.oql>. Running a .oql file is a live query execution against omega-db. The side effects of that execution (e.g. writing clause definitions to the database) persist. There is no separate "upload" step — running the query IS the deployment.

qo push does NOT work for .oql files. qo push expects a JS component implementation with an impl/index.json index. Pointing it at a .oql file will fail or produce incorrect behavior.

OQL component implementations (JS files, not .oql)

A component implementation is a JavaScript file produced by qo pull. It contains an embedded OQL script (stored as a string literal) alongside a metadata index (impl/index.json). The OQL inside the impl file is executed at runtime by the Omega engine when the component is invoked — it is not executed at deploy time.

How they are deployed: via qo push [impl-file]. qo push reads impl/index.json, finds the component-id and impl-id, and uploads the JS implementation to the workspace. The OQL inside is not run at push time — it is stored for later runtime execution.

Key distinction: A component implementation file uses OQL internally, but it is not an OQL file. It is a JS file. Running omega-cli run-query on a JS impl file would fail — that command expects a .oql file.

The two CLIs

omega-cli — database and library access

Binary: omega-cli

This CLI provides direct access to omega-db. The primary OQL-relevant command is:

  • omega-cli run-query <file.oql> — executes an OQL file as a live query against omega-db. Use this to deploy library clauses, run scratch queries, and execute backfill/migration queries.

omega-cli is the lower-level tool. It talks directly to the database and is the correct tool for any task involving .oql files.

qo (query-omega-cli) — application-layer operations

Binary: qo

This CLI manages application-layer Omega objects: pages, components, implementations, and packages. Key commands relevant to OQL development:

  • qo pull [component-path] — downloads a component implementation from the workspace into a local JS file with impl/index.json.
  • qo push [impl-file] — uploads a JS component implementation to the workspace. Requires impl/index.json. Does NOT accept .oql files.
  • qo run <query-path> -p <page> -m run -a '[{...}]' — runs a named query from a workspace page.
  • qo doc push <pages-dir> "Docs/Section" — syncs HTML docs to the OmegaAI workspace.

qo is the higher-level tool. It knows about workspace structure, page hierarchies, and component lifecycles. It does NOT know about raw .oql files.

Decision table

Task Correct tool Wrong tool
Deploy a library clause (.oql file) omega-cli run-query file.oql qo push — will fail
Run a scratch query omega-cli run-query scratch/query.oql qo run — wrong API surface
Push a JS component implementation qo push impl/index.json omega-cli run-query — wrong file type
Pull a component from the workspace qo pull <path> n/a
Run a named workspace query qo run <page/query> ... omega-cli run-query — different concept
Sync HTML docs to workspace qo doc push pages/oql "Docs/OQL-Public" n/a

The lifecycle in one sentence per artifact type

OQL library clause: Write a .oql file → omega-cli run-query file.oql → clause definitions stored in omega-db → callable by any subsequent OQL query.

OQL component implementation: qo pull <path> → edit the embedded OQL string in the JS file → qo push impl/index.json → new OQL stored as component impl → executed at runtime when the component is invoked.
