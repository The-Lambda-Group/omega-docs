Scratch Queries

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

Scratch queries are bare top-level term sequences with no clause mechanism. They are run with qo run or qo query. Scratch queries are read-only — no mutations.

Always end with (return [Result])

Unlike implementation files (where clauses return via the head binding), scratch queries have no implicit return path. If you bind Result but never call (return [Result]), the query runner produces no result and the API returns a bare HTTP 500 with no error message, no stack trace, and nothing actionable.

;; WRONG — bare 500, no diagnostics
(= Result "hello")

;; RIGHT
(= Result "hello")
(return [Result])

The 500 with no body is the only signal that (return [...]) is missing. There is no other indication. Memorize this pattern.

Scratch queries are read-only

Scratch files have no review trail, no implementation identity, and no protocol scoping. A mutation that lands from a scratch file is untraceable. The rules:

  • No !-suffixed terms that write to datastores.
  • No delete!, refresh, or any write operation.
  • No write-table, set-data, or similar mutations.

Scratch queries are for read-only inspection, returning bound values for dry-run verification, and building up an understanding of the data shape before writing an implementation.

run-term in scratch file top-level fires once — not per solution

run-term in the top-level body of a scratch file fires once regardless of how many solutions are in scope from a preceding generator like prop-vals-by-folder. It does not iterate over all solutions the way it does inside a clause body.

Wrong pattern (only processes one row):

(prop-vals-by-folder FolderId RowPid RowPv)   ;; N solutions
(get-in RowPv ["prop-vals" "content"] Content)
(run-term MyHelper RowPid Content Done)        ;; fires ONCE, not N times
(fold [] Done append Dones)                    ;; Dones = ["ok"], length 1

Correct pattern — collect into list first, iterate inside a clause body:

;; Outer body: collect all inputs into a list (single solution after fold)
(prop-vals-by-folder FolderId RowPid RowPv)
(get-in RowPv ["prop-vals" "content"] Content)
(= Pair [RowPid Content])
(fold [] Pair append Pairs)

;; Clause body: iterate the list, run-term fires N times inside here
(:- (ProcessAll Pairs Done {"capture" [MyHelper]})
    (with-table-if (get Pairs 0 _)
      [Pairs MyHelper Done]
      (and (get Pairs _ Pair)
           (get Pair 0 RowPid)
           (get Pair 1 Content)
           (run-term MyHelper RowPid Content ItemDone)
           (fold [] ItemDone append _)
           (= Done "ok"))
      (= Done "skipped")))

(call ProcessAll Pairs _)
(return [Result])

Calling a !-suffixed clause from a scratch file

Using (Ds/term! Args) in a scratch query body invokes the write pathway. If Args contains unbound symbols (which it almost always does in a scratch query), the engine throws WRITING_SYMBOLS. The read/write distinction is at the call site:

(Ds/term Args)   ;; READ — queries the stored clause, produces solutions
(Ds/term! Args)  ;; WRITE — stores a clause (not what you want in a scratch query)

Typical scratch query pattern

;; Look up pages by folder
(Qo.Public.OqlApi.Page/pages-by-folder FolderId Page)
(get Page "page-id" PageId)
(get Page "page-name" Name)
(= Result {"page-id" PageId "name" Name})
(return [Result])

Inspecting intermediate state

When a scratch query produces unexpected results, return intermediate state to inspect:

;; Return intermediate values as a tuple for inspection
(Qo.Public.OqlApi.Db.Prop/prop-vals-by-folder FolderId PageId PropVals)
(= Result [PageId PropVals])
(return [Result])

You can bind Result to any combination of symbols. The (return [...]) call accepts a vector of whatever you want to surface.

Count rows in a defterm (without OQL)

The recommended approach for pre-migration audits and sanity checks — it does not require a running OQL query.

Every defterm-backed datastore is a CouchDB database. CouchDB maintains doc_count as O(1) view metadata.

Step 1 — Find the CouchDB database name:

omega-cli db-name <datastore-path>

Use the datastore path (the string in the datastore declaration, not the OQL namespace symbol). For example:

omega-cli db-name "omega/query-omega/data/log-stream"
# Output: omega_query-omega_data_log-stream

Step 2 — Read doc_count from CouchDB:

curl -s http://admin:password@localhost:5984/<db-name>/ | jq '.doc_count'

This stays fully outside OQL and outside Rule 2's scope (no direct Qo.Data.*/ access). The all-wildcard form of with-count — (with-count Cnt (Qo.Namespace/term _ _ _ _)) — currently fails with omega/jvm/NullPointerException, so this CouchDB approach is the only safe way to get a total row count.
