This is the manual for working in OQL. Read it before touching a .oql file. It covers the mindset both authoring and debugging depend on, the inspection mechanic the language gives you in place of print statements, and the loop you operate forward (when adding) or backward (when debugging). Authoring and triage are the same loop in two directions; they share this doc.

Every plan that touches OQL implementation work MUST list this doc as the single most important doc and require the implementer to read it before any code change. This rule exists because every prior multi-session OQL build that produced a discipline failure shared one root cause — the implementer hadn't internalized this doc.

Operating manual (re-read every session)

These rules are what to actually do, in order. The rest of the doc is rationale. If you're an agent and you only have time for one thing, read this section.

  1. Each run answers exactly one question. Before changing the file, name the question out loud: "what's bound to X right now?", "does this fold collapse?", "what shape does write-table reject?" If you can't name the question, you're in build-mode. Stop and name it.
  2. Inspect by mutating Result. Comment out everything after the term you want to look at. Bind Result to the variable, list, or map you want to surface. Run. Read the output. There are no print statements in OQL — Result IS the inspection channel.
  3. When a term fails, comment it out and bind Result to its inputs. The failing term told you that it failed. Its inputs tell you what data it failed on. Don't theorize; look at the inputs. The bug is almost always visible there.
  4. When wall-clock grows past 1–2 seconds for a small workload, you have a cardinality leak. Don't add features — find the leak. Common shapes: an unfolded (get list _ var) interacting with another (get otherlist _ othervar) (cross-product), a captured-helper invocation that opens N solutions when you thought it returned one, a with-group-by over many distinct keys.
  5. If you catch yourself writing "I think the issue is…" or "this is probably because…", stop. That's the theorize-mode signal. Close the file, redirect Result to the relevant intermediate, run. The state will tell you. Theories are post-hoc and disposable; observations decide the next probe.

The push-run loop

OQL development is REPL-driven. You build queries incrementally, one step at a time, verifying each step before adding the next. The push-run cycle is fast enough to test every single change — there is no reason to batch work.

Before writing any OQL implementation, read:

  • OQL hard rules — non-negotiable rules on forbidden datastores and the boolean-string requirement.
  • Execution model — how OQL evaluates queries (solution sets, why terms can't be nested inside literals).
  • Built-in terms — the full built-in term reference.
  • Control flow — read this before writing any when, when-not, if, or with-table-if.

Debug by returning early

When a query fails partway through — wrong HTTP status, missing row, mysterious unbound symbol — the fastest path to the root cause is to comment out everything after the suspected failure point and return a snapshot of the intermediate state. Re-run the query, inspect the snapshot, then un-comment one more clause and repeat.

The recipe:

  1. Find the clause you suspect is broken.
  2. Comment out every clause after it (including the original Result binding).
  3. Bind Result to a tuple of the variables you want to inspect: (= Result [Var1 Var2 Var3])
  4. Run. Observe. Move the return point forward one step. Repeat.

This is not just for debugging. It's also the authoring loop — build one step, verify, build the next. The loop is the same in both directions.

The Result-as-inspection-channel rule

OQL has no print statements and no debug logging from the query layer. The only way to see what a term is binding is to make Result be the thing you want to see. This means:

  • During development, Result is a diagnostic variable as much as it is the final output.
  • The final (return [Result]) is the only output channel. Use it for inspection at every intermediate step.
  • Binding Result to [X Y Z] returns the tuple — you can inspect multiple values at once.

One change at a time

When debugging, make exactly one change per push-run cycle. If you make multiple changes and the bug is fixed, you don't know which change fixed it. If you make multiple changes and the bug persists, you've mixed signals. The discipline: one question per run, one change per push, one observation per cycle.

The same discipline in build mode: add one term or clause at a time, verify it produces the expected bindings, then add the next. Don't write 15 terms and push — that's build-mode. Push at each term.

Cardinality as a diagnostic signal

If your query runs for more than 1–2 seconds on a small workload, you have a cardinality explosion. The most common shapes:

  • Cross-product: two (get list _ var) terms in the same clause body without a fold between them. Each produces N solutions; the cross-product is N×M.
  • Captured helper opens multiple solutions: a helper invoked via call produces more than one solution row per input. The caller's solution table multiplies by the helper's cardinality per call.
  • with-group-by over many distinct keys: each key triggers a separate accumulation pass. Switch to run-term per key for the partition-before-fold pattern.
  • with-table-if condition is a multi-row scan: the then-branch runs once per condition row, not once total. See the existence-check fallacy in Control Flow.

Diagnose cardinality by returning early and counting rows: (= Result [X (length Y Len) Len]) where Y is the suspected cross-product.

Verb shift for agents

Two mindsets that are incompatible with good OQL work:

  • Build-mode: writing many terms in a batch, pushing once, reading the error, theorizing a fix, writing more terms. Build-mode is slow, error-prone, and produces code that is hard to debug. The antidote is the one-question-per-run rule.
  • Mirror mode: agreeing with whatever the last output said and restating it as the next move. Mirror mode produces a chain of confirmed mistakes. The antidote is to redirect Result to the inputs of the failing term before any theory.

The verb for OQL work is probe, not build. You probe the current state. You observe what comes back. You make the minimal change that answers the next question. You probe again.

Floundering signals

If you observe any of these, stop immediately and surface to the user:

  • The same error appearing after the same fix has been applied twice.
  • Three different errors in one session with no consistent trend.
  • A wall-clock longer than 30 seconds on any operation that should complete in seconds.
  • A socket hang-up or ECONNRESET — the engine may be stalled and require a pod restart.
  • A "not sure what to try next" state without a concrete next probe defined.
