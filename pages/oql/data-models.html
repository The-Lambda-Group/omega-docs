OmegaDB Core Type Shapes

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

This page documents the exact stored field shapes for the five core OmegaDB types: Component, Protocol, Implementation, Package, and Page. Use this reference when writing OQL that reads from or writes to these types. Field names are case-sensitive JSON string keys.

How to read this page

Each type section shows (1) the datastore functor definition from source, (2) the canonical stored body shape, and (3) notes on which fields agents commonly misread. The body map is what you get back from a lookup clause — e.g. (Qo.Comp/component AppId CompId Comp) binds Comp to the body map.

Component

Source: query-omega-oql/oql/app/component.oql, query-omega-api/oql/app/public/oql-api/component.oql

Datastore: Qo.Data.Comp (omega/query-omega/data/component), functor comp(AppId, CompId, Comp), primary key CompId.

Lookup: (Qo.Comp/component AppId CompId Comp)

Stored body (Comp):

{
  "type"         : "omega/query-omega/component/Component",
  "name"         : <string>,
  "app-id"       : <uuid string>,
  "page-id"      : <uuid string>,
  "component-id" : <uuid string>
}

Note: page-id here is the page the component was created on (its "home" page). Components do not have an implementation-id field — implementations are separate objects linked by component-id. Components may also carry an optional description field written by update-description!.

Protocol

Source: query-omega-oql/oql/app/protocol.oql, query-omega-api/oql/app/public/oql-api/protocol.oql

Datastore: Qo.Data.Proto (omega/query-omega/data/protocol), functor proto(AppId, ProtoId, Proto), primary key ProtoId.

Lookup: (Qo.Proto/protocol AppId ProtoId Proto)

Stored body (Proto):

{
  "type"        : "omega/query-omega/protocol/Protocol",
  "name"        : <string>,
  "app-id"      : <uuid string>,
  "page-id"     : <uuid string>,
  "protocol-id" : <uuid string>,
  "spec"        : <map — protocol argument spec>
}

Note: page-id is the page the protocol was created on. spec is the protocol argument specification map passed at creation time via add-or-get-by-name!.

Implementation

Source: query-omega-oql/oql/app/implementation.oql, query-omega-api/oql/app/public/oql-api/implementation/implementation.oql

Datastore: Qo.Data.Impl (omega/query-omega/data/implementation), functor impl(AppId, ImplId, CompId, ProtocolId, Impl), primary key ImplId.

Lookup: (Qo.Impl/implementation AppId ImplId CompId ProtoId Impl)

Stored body (Impl):

{
  "type"              : "omega/query-omega/implementation/Implementation",
  "app-id"            : <uuid string>,
  "code"              : <string — OQL source text or other language source>,
  "language"          : <string — e.g. "oql">,
  "component-id"      : <uuid string>,
  "protocol-id"       : <uuid string>,
  "implementation-id" : <uuid string>
}

Critical note: Implementation does NOT have a page-id field. Implementations are linked to a page indirectly through their component and through the page-block that references the implementation-id. Attempting to read (get Impl "page-id" PageId) will fail silently (no solutions). To find the page for an implementation, look up the page-block with the matching implementation-id using (Qo.Page.Bl/page-block _ PageId _ Block) and (get Block "implementation-id" ImplId).

A second stored type, ImplementationClauses (Qo.Data.Impl.Clauses), holds parsed clause data separately: {"type": "omega/query-omega/implementation/ImplementationClauses", "implementation-id": ..., "clauses": [...]}. It is read/written independently from the main Impl body.

Package

Source: query-omega-oql/oql/app/package.oql, query-omega-api/oql/app/public/api/package.oql

Datastore: Qo.Data.Pkg (omega/query-omega/data/package), functor package(AppId, PackageId, Package), primary key PackageId.

Lookup: (Qo.Pkg/package AppId PackageId Package)

Stored body (Package):

{
  "name"       : <string>,
  "app-id"     : <uuid string>,
  "package-id" : <uuid string>,
  "page-id"    : <uuid string>
}

Note: page-id is the page on which the package was created. The get-package! API also injects a virtual "docs-page-id" field from the PackageDocs table when present (falls back to "none"), but this field is not stored in the Package body itself.

Related types: component-package (links a component to a package, keyed by [CompId, PackageId]) and package-install (records an installation in an app, keyed by [AppId, PackageId]). Neither is part of the core Package body.

Page

Source: query-omega-oql/oql/app/page.oql, query-omega-api/oql/app/public/api/page.oql

Datastore: Qo.Data.Page.Data (omega/query-omega/data/page), functor page(AppId, FolderId, PageId, Page), primary key PageId.

Lookup: (Qo.Page/page-object AppId FolderId PageId Page) — or (Qo.Page/page AppId FolderId PageId PageData) for the raw page-data only.

Stored body (Page):

{
  "type"      : "omega/query-omega/page/Page",
  "app-id"    : <uuid string>,
  "folder-id" : <uuid string — parent page's page-id, or root folder id>,
  "page-id"   : <uuid string>,
  "page-data" : {
    "name"   : <string>,
    "blocks" : <array — present for content pages; absent for database pages>
               OR
    "is-database" : "true"
  }
}

Note: folder-id is the parent container's id — for top-level pages this is the root folder id, for nested pages it is the parent page's page-id. The page-data.blocks array holds inline block content for legacy/content pages. Database pages use "is-database": "true" instead. The page-lookup! helper additionally injects "component-id" from the page-component join, but this is computed, not stored in the Page row.

Defterm storage layout

All five types use the same CouchDB defterm pattern: the body map is stored as args[N] in the underlying document where N is the last positional argument. The preceding args (AppId, FolderId, etc.) are stored as args[0], args[1], etc. and form the basis for write-index design docs. When writing Mango queries directly against these datastores, use "args.N.field-name" selector syntax — e.g. "args.2.name" for component name search.
