LLM Output Discipline

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

When an LLM-driven agent step pipes a model response into OQL — typically through (json-stringify Parsed RawText) to land structured data into the solution set — the response sometimes won't parse. The reflexive instinct is to add a string-replace or regex-clean pass before the json-stringify call, normalising the malformed bytes into something the parser will accept.

Don't.

The rule

If an LLM response can't be parsed by OQL's json-stringify:

  1. Do not insert a pre-parse cleaning step (string-replace, regex normalisation, character stripping, etc.) into the implementation.
  2. Do fix the producer — tighten the LLM prompt so the model emits parser-clean output the first time.
  3. Or file a parser-bug gap if the producer can't reasonably be tightened, and stop. Don't ship a workaround in the meantime.

Two acceptable moves, both at the boundary that owns the data shape. No scrubbing layer in between.

Why string-scrubbing is the wrong response

A scrubbing layer at the consumption boundary:

  • Hides the real failure. If the LLM is emitting malformed output, the scrubber masks the signal. The next version of the prompt, the next LLM, or the next deployment removes the scrubber and the failure returns — but now it's harder to diagnose because it was hidden for so long.
  • Silently corrupts data. A string-replace that removes a character the parser rejects might also remove that character from legitimate content — stripping a Unicode character from an LLM response that legitimately contained it, for example. The scrubber is a lossy transform applied without semantic understanding of the content.
  • Creates a maintenance liability. The scrubber needs to handle every future variant of the failure. As the LLM changes and the prompt evolves, the scrubber becomes an accumulating layer of special cases with no clear exit.
  • Is technically incorrect. The LLM is the producer. The producer owns the shape of its output. Normalizing at the consumer boundary is a violation of the producer/consumer contract — it says "I accept any output and will make it work," which removes the incentive for the producer to emit correct output.

What json-stringify parse direction is strict about

json-stringify parse direction (first arg unbound, second arg bound) rejects raw control characters — ASCII 0–31 other than space (0x20), newline (0x0A), and tab (0x09) — that are illegal in JSON per RFC 8259. An LLM response that contains these characters as raw bytes in a string value is technically invalid JSON, and json-stringify correctly rejects it.

The error message is:

omega/jvm/Exception: JSON error (missing entry in object)

or:

omega/jvm/Exception: JSON error (unexpected character)

This is not a parser bug. The parser is correct. The JSON is invalid.

LLM response truncation

A call-llm invocation fails with:

omega/jvm/EOFException: JSON error (end-of-file inside string)

This looks like a JSON parse bug. It is not. The LLM emitted a response that was cut off mid-string (before the closing braces and quotes that would make the payload valid JSON), so json-stringify runs out of input while still inside an open string and throws EOFException. Wall-clock to the throw is typically 30s-2min — long enough that the LLM was clearly generating output, not failing fast on input.

The cause is always prompt design or token limit:

  • The LLM was asked to produce a JSON payload that exceeds the model's output token limit.
  • The prompt didn't instruct the model to keep responses concise.
  • The model was generating verbose reasoning or preamble before the JSON, consuming tokens that should have gone to the JSON payload.

The fix is always at the prompt:

  • Add an explicit instruction to return only the JSON object, no preamble or explanation.
  • Reduce the expected output size (fewer fields, shorter values, paginated output).
  • Increase the model's output token limit if available.

Do not add retry logic or partial-parse recovery at the consumption layer. The problem is the producer, not the consumer.

When to file a parser-bug gap

If you have tightened the prompt as much as is reasonably possible and the LLM still emits output that json-stringify rejects — and you have evidence that the output is technically valid JSON that the parser should accept — file a gap. Describe:

  • The exact JSON that was rejected (or a minimal repro).
  • The exact error message.
  • Evidence that the JSON is technically valid (e.g., it parses in other JSON parsers).
  • Why prompt-tightening is not sufficient to address the failure.

Stop at the gap. Do not ship a scrubber while the gap is open.
