What is Omega?

Omega is a notebook-based application platform backed by a logic database. In Omega, everything is a page. A page can hold code, data, configuration, and logs — just like a cell in a notebook can hold different kinds of content. Pages are organized into trees, and the whole system is queryable and programmable through OQL (Omega Query Language), a datalog-style logic language.

The Notebook

The core of Omega is a notebook. You create pages, add blocks to them, and build applications by composing pages together. The notebook runs in a browser, but you can also interact with it entirely from the command line using the qo CLI.

A page can have several kinds of blocks:

  • Code blocks — OQL implementations that define behavior
  • Data blocks — database tables with rows, properties, indexes
  • Config blocks — push connectors that store key-value configuration (like environment variables)
  • Log blocks — log streams for observability

When you write code in a page's code block and save it, the code is evaluated immediately — clause definitions are stored in the database, ready to be invoked later. This is how Omega works: code and data live together in the same system.

Pages

Everything in Omega lives on a page. A page is a container that can hold code, data, configuration, and logs. Pages are organized into a tree — every page has a parent, and can have children.

The page tree is how Omega organizes an application:

My App/
  Event Listener/          <- has a log stream, receives events
  Events/
    Coordinator/           <- code block: routes events to steps
    Score Agents/          <- code block: scoring logic
    Filter Agents/         <- code block: filtering logic
  Data/
    Scored Agents/         <- data block: database table
    Filtered Agents/       <- data block: database table

Agent-readable page trees

qo doc push creates bare folder pages — folder nodes with no content, just a name and children. An agent running qo ls "Docs/MAB" sees child names but has no signal about what is in each folder without descending blind.

The fix: a README.html file at every directory level in the pages/<project>/ source tree. When pushed to OmegaAI, each README.html becomes a doc page named README, a child of its parent folder. Agents navigating the doc tree use these README pages as navigation roots — reading the folder summary before deciding which child to descend into. This eliminates the need to exhaustively list every subtree to answer a question.

OQL as the logic layer

OQL is a logic programming language inspired by Datalog and Prolog. Instead of writing step-by-step procedures, you describe relationships and constraints, and the database engine finds all solutions that satisfy them.

The core insight: all computation is ultimately relational, not operational. A relation is the algebra of logic itself: AND, OR, NOT over data. Functions are a special case of relations — they constrain the output to exactly one value per input. Relations are the general case.

Imperative code encodes two things fused together: the logic (what you want) and the execution strategy (how to get it). Relational code encodes only the logic. The execution strategy is left to the engine. This means imperative code contains accidental complexity by definition — every for loop, every if chain, every callback is an execution decision that has nothing to do with the problem being solved.

OQL dissolves this. A relation says "these things are true together." It doesn't say in what order to check them, how to traverse the data, or what to do first. It is a pure statement of what is true.

OQL language fundamentals

The basic unit of OQL is a term — a statement that is either true or false.

(blue "Sky")           ;; "Sky" is blue
(even 2)               ;; 2 is even
(- 6 1 5)              ;; 6 - 1 = 5

Capitalized names are symbols (variables). The engine tries to find values for symbols that make all terms true simultaneously. This is unification — the engine doesn't assign variables top-down like imperative code. It finds consistent bindings across all terms.

OQL works natively with JSON-like maps:

;; Bind Name to whatever value is at "name"
(get Page "name" Name)

;; Constrain — only succeeds if name is "Score Agents"
(get Page "name" "Score Agents")

;; Nested access
(get-in Page ["page-data" "name"] Name)

Unification applies to every term. When you write (get Map "key" Value), the third argument is unified — if Value is unbound, it binds to whatever the key holds. If Value is a literal, the term only succeeds when the key's value matches that literal. This matters for control flow: (if Condition Then Else) checks whether Condition succeeds as a term, not whether it returns a truthy value.
