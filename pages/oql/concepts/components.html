Components, Protocols, and Implementations

Omega's code model has three parts: components, protocols, and implementations. Together they define what a page can do and how it does it.

Protocols

A protocol is an interface definition. It declares a set of methods with names and arities (argument counts). Protocols don't contain any code — they just describe what methods exist.

{
  "functions": [
    { "name": "install", "args": ["self"] },
    { "name": "start",   "args": ["self"] },
    { "name": "stop",    "args": ["self"] },
    { "name": "status",  "args": ["self"] }
  ]
}

Protocols are the interfaces; components are the types; implementations are the code. See Protocols reference for the full list of standard protocols.

Components

A component is a type definition. It says "this page is capable of implementing these protocols." A component can implement multiple protocols.

Implementations

An implementation is the actual OQL code that fulfills a protocol's methods for a specific component. Each method is an OQL clause that takes Exec and Result as its two arguments:

(:- (MyInstall Exec Result)
    ;; set up pages, databases, subscriptions
    (= Result {"status" "ok"}))

Implementations are registered via set-implementation-clauses. The call maps method names to OQL clause symbols:

(Qo.Public.OqlApi.Impl/set-implementation-clauses
  "$$ARG$$"
  {"install" MyInstall
   "start"   MyStart
   "stop"    MyStop}
  _)

Plugins

A plugin is an installable component that creates its own page tree in your workspace. Think of it like apt-get install — you install a plugin, it sets up everything it needs, and you operate it with standard tools.

What a plugin creates when installed

Component Installs/
  Payments Connector/               <- the plugin's root page
    Payments Event Listener/        <- event handler + log stream
    Payment Events/                 <- source data table (raw webhook events)
    Events/
      Coordinator/                  <- routes events between steps
      Parse Events/                 <- event parsing logic
      Reconcile Balances/           <- reconciliation logic
      Report Deltas/                <- reporting logic

The install method of the Installable protocol is responsible for creating this structure. A well-written install method creates the page tree, initializes databases, registers subscriptions, and sets up push connectors — all in a single idempotent pass.

Idempotent install

Install methods should be idempotent — running install twice should produce the same result as running it once. Use add-or-get-* API clauses rather than add-* to achieve this:

;; Idempotent page creation
(Qo.Public.OqlApi.Page/add-or-get-sub-page-by-name ParentPage "Events" EventsPage)

;; Idempotent database creation
(Qo.Public.OqlApi.Page/add-or-get-sub-database-by-name ParentPage "Treatments" TreatmentsDb)

The page tree convention for workflow components

Workflow components follow a standard page tree convention:

My App/                                  <- root: Installable + Service
  My App Event Listener/                 <- Event Handler block
  Events/
    Coordinator/                         <- Event Handler: routes to steps
    Step A/                              <- Event Handler: one step
    Step B/                              <- Event Handler: another step
  Data/
    Step A Results/                      <- database table
    Step B Results/                      <- database table

The Coordinator's on-event method receives events from the Event Listener and routes them to the appropriate step via run-page. Each step's on-event method does the actual work and writes results to the Data table.

Event subscriptions and handlers

When an event subscription handler fires, it creates a fresh context — it does NOT inherit any state from the original emitter. The handler functor must be fully self-contained.

You cannot pass runtime state through subscriptions — only through the event data payload. Design your event payloads to include everything the handler needs to do its work.

Run clauses

The Query protocol's run method enables stateless query execution. run-page and run-page-by-id invoke a page's run method:

;; Run by page reference
(Qo.Public.Api.Run/run-page Proto Page Name Args Result)

;; Run by page ID
(Qo.Public.Api.Run/run-page-by-id PageId ProtoName Name Args Result)

Use run-page-by-id when you have the page ID cached (faster — avoids a page lookup). Use run-page when you only have the page object.
