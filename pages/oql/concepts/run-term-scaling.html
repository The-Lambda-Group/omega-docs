run-term Scaling

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

The rule

For in-memory captured-helper invocations: use call by default. run-term is appropriate ONLY in two specific contexts:

  1. Partition-before-fold — replacing with-group-by with per-partition run-term invocations.
  2. Per-row iteration at scale — a helper invoked once per row in a multi-row solution table where K ≥ ~30 rows.

In every other context — one-solution-in / one-solution-out helpers, list-mapping helpers, validation helpers, transformation helpers — (run-term ...) against an in-memory captured helper is wrong. The hard failure mode is a 500. The softer, more dangerous failure mode is engine spin-out / ECONNRESET — a malformed run-term invocation can stall the database and require a pod restart.

If you are not replacing a with-group-by and not in a per-row iteration at scale, you are not in the exception. Use call.

Why agents get this wrong

The documentation for run-term covers both the partition-before-fold case and the per-row-iteration case. Agents reading the performance section of the reducers doc often conclude "use run-term for performance" and apply it to all captured-helper calls. This is wrong. run-term is not a general performance optimization — it is the correct tool for exactly two shapes, and it actively breaks other shapes.

Why run-term works in the two valid shapes

run-term runs its target as a separate execution against a fresh solution. This is mechanically what makes the partition-before-fold pattern fast (each invocation sees only one partition's rows) and what makes the per-row iteration pattern work at scale (each invocation sees only one row's bindings, preventing the K-row solution-table residue that causes call to hang).

Outside these two shapes, the "fresh solution per invocation" mechanism is not needed and produces the wrong result or fails outright.

Partition-before-fold scaling

fold / with-group-by / with-order-by scale poorly when the solution table contains many distinct group-key values. Per-partition bookkeeping compounds non-linearly with partition count.

The idiomatic workaround: wrap the reducer body in a clause and invoke it once per key via run-term. Each invocation's solution is single-partition, so the reducer chain runs against only that partition's rows.

Measured 2026-04-24: 8.3s with a top-level with-group-by [PageId], 0.5s when the same reducer body was called via run-term per page (21-page pull).

Per-row iteration at scale

A per-row iteration is any pattern where the solution table has K rows and a helper is invoked once per row, with per-row input bindings:

;; K rows in scope (one per arm)
(get Arms _ ArmData)
(get ArmData "arm_id" ArmId)
(get ArmData "budget" Budget)

;; Helper invoked once per row — use run-term at K=40
(run-term BuildAllocationsRow Ctx ArmId Budget Row)   ;; CORRECT at K=40

;; BAD at scale — hangs at K~40
;; (call BuildAllocationsRow Ctx ArmId Budget Row)

Empirical evidence

Two independent cases from MAB Allocate V1 (2026-04-30):

  • BuildAllocationsRow: original call at K=40 produced 3 socket hangs in one debug session. Fix to run-term brought wall-clock to ~3–5s.
  • CountSaForArm: original call at K=40 caused the engine to hang and require a database restart. Fix to run-term brought wall-clock to <5s.

Empirical threshold: call in per-row iteration hangs somewhere between K=27 and K=40. The exact threshold is unknown. At K=40 it reliably hangs; K=27 has not been tested in the iteration-specific shape.

call vs run-term decision table

ContextUse
One-solution-in / one-solution-out helper (standard shape)call
List-mapping, validation, transformation helper (not per-row iteration)call
Partition-before-fold scaling (replacing with-group-by)run-term
Per-row iteration at scale (K ≥ ~30 rows)run-term

What to do when you hit a hang

  1. Identify the per-row helper invocation(s) in the hanging clause.
  2. Switch (call Helper ...) → (run-term Helper ...) for each per-row invocation.
  3. Verify that the helper's logic is correct for stateless per-invocation execution (no side-effects that depend on cross-row shared state — each run-term call is independent).
  4. If the engine is stalled after a hang: restart the omega-db pod.

Hypothesis on root cause

(call CapturedHelper ...) per-row in a multi-solution iteration may produce solution-table residue that scales non-linearly. Each call invocation shares solution scope with the enclosing iteration — the K-row solution table is visible during each call. As K grows, the accumulated per-call scope overhead may compound.

(run-term CapturedHelper ...) runs the helper as a separate execution against a fresh solution. This isolates per-row state from the enclosing K-row iteration scope. This is the same mechanism that makes run-term fast in the partition-before-fold pattern — the key insight is that "fresh solution per invocation" is beneficial in both cases.

This hypothesis is not empirically verified — it is the best current explanation for the observed behavior.
