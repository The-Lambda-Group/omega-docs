S-Expression Evaluation Model

The core rule: S-expressions extend solution bindings

OQL S-expressions do not produce values that can be used inline at the call site. Every S-expression extends the current solution set by adding new symbol bindings. A term's output is a new symbol in scope — never an inline expression result.

This is the fundamental difference from Clojure, Lisp, or any value-expression language. In those languages, (get map "key") returns a value you can embed directly. In OQL, (get Map "key" Result) binds the symbol Result into the current solution row. The term succeeds or fails — it does not "return" anything.

Consequence 1: map literals require pre-bound symbols

Map literals in OQL accept only already-bound symbols as values. You cannot call a term inline inside a map literal {}.

;; WRONG — silently produces an unbound value for "page-id"
{"page-id" (get Page "page-id") "name" Name}

;; CORRECT — bind first, then use the symbol
(get Page "page-id" PageId)
{"page-id" PageId "name" Name}

The wrong form does not throw an error — it silently produces an unbound value for that key. The map literal syntax does not evaluate nested S-expressions; it reads already-bound symbols only. The same rule applies to any nested data structure: map literals, vector literals, and argument positions in other terms all require pre-bound symbols, not inline term calls.

Consequence 2: terms run once per clause entry, shared across all solution rows

When a term appears at the top level of a clause body, it runs once when the clause is entered. Its result binding is shared across every solution row that flows through the clause. For terms that produce a stable value (like (get Map "key" Val) on a single bound row) this is fine. For terms that generate fresh values on each call — like uuid — this causes a single value to be reused across all rows.

;; WRONG — one UUID shared across all rows → KEY_UPDATE_CONFLICT
(uuid BlockId)
(write-table SomeTableId
  [{"block-id" BlockId "content" Content}]
  _)

In the wrong form, every row in the write batch gets the same BlockId because uuid ran once, extended the solution with one binding, and that binding is reused for every row.

Fix: use run-term to execute a helper once per solution row. run-term creates a fresh solution partition for each row, so uuid runs independently for each:

;; CORRECT — uuid called once per row via run-term
(:- (BuildBlockRow Content BlockId)
    (uuid BlockId))

(run-term BuildBlockRow Content BlockId)
(write-table SomeTableId
  [{"block-id" BlockId "content" Content}]
  _)

The mental model

Think of OQL clause execution as a pipeline of solution tables (see OQL Execution as SQL Joins). Each term is a JOIN operation against the current solution table. The JOIN adds new columns (new symbol bindings) and may filter rows. It never "returns" a value — it transforms the table.

Map literals, vector literals, and term arguments read from the current solution table's columns. They cannot trigger a JOIN mid-expression. Any computation that produces a new binding must be a separate S-expression (a separate JOIN step) before the literal that reads it.

Symbols-in-literal-arguments rule

The same principle applies everywhere, not just map literals. For the canonical statement and more examples of valid vs invalid argument positions, see Built-in Terms — Symbols in Literal Arguments.
