Query Execution & Index Selection

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

This document covers how the OQL engine selects an execution strategy when evaluating a term query, and how with-sort, with-skip, and with-limit behave under each strategy. Understanding this is essential for avoiding silent correctness bugs where pagination parameters are silently ignored.

The four strategies

Strategy 1 — Defterm PK lookup

When it fires: the bound columns exactly match the term's declared primary key.

Mechanism: the engine computes deterministic row ids from the PK values and fetches them in a single _all_docs POST.

OperatorPushed?
with-limitNo
with-skipNo
with-sortNo

Strategy 2a — Exact write-index bind

When it fires: a write-index covers the bound positions, AND the number of bound positions equals the total number of fields in the index (every index field is bound).

Mechanism: a single POST with {:keys ks} against the view.

OperatorPushed?
with-limitYes
with-skipYes
with-sort "desc"Yes

This is the highest-performance path and the only one where all three operators are fully pushed down.

Strategy 2b — Prefix bind, one bound row

When it fires: a write-index covers the bound positions as a prefix AND there is exactly one bound row.

Mechanism: a range query using startkey / endkey. The endkey sentinel is (conj (vec bound-key) {}) — an empty map sorts after all scalar values in CouchDB's collation order, closing the range.

OperatorPushed?
with-limitYes
with-skipYes
with-sortYes

Strategy 2c — Prefix bind, N bound rows

When it fires: a write-index covers the bound positions as a prefix AND there are two or more distinct bound rows.

Mechanism: one lazy-range-fetch per bound row, merged via a k-way merge heap (omega-db.datastore.couchdb.heap-merge/k-way-merge). Memory cost is O(K) where K is the number of ranges.

OperatorPushed?
with-limitNo — applied AFTER merge
with-skipNo — applied AFTER merge
with-sortPartially — drives the merge key-fn, not pushed to individual view requests

If a query binds multiple values across two dimensions and a secondary index covers only the first dimension as a prefix, OQL will silently issue one range fetch per outer-dimension value and merge them in memory. with-limit 10 will still return only 10 rows, but the database may scan far more.

Strategy 3 — Mango fallback

When it fires: no write-index covers the bound positions as a prefix AND (enable-full-scan) is active.

Mechanism: a CouchDB Mango query against the raw document store.

OperatorPushed?
with-limitNo
with-skipNo
with-sortNo

The Mango path returns up to the page-query default ceiling (effectively unbounded). with-limit and with-skip are silently ignored — the query returns all matching rows with no error or warning. The page-query default limit is 1,000,000 — effectively unbounded. If a caller passes with-limit 10 but hits the Mango fallback, the query quietly returns up to one million rows instead of ten.

Strategy 4 — No-matching-index error

When it fires: no write-index covers the bound positions AND enable-full-scan is NOT active.

Mechanism: the engine throws immediately:

{:type "omega/query/no-matching-index"
 :bound-positions [...]
 :candidate-indexes [...]}

This is the safest outcome — it surfaces the problem explicitly rather than silently scanning.

Note: the Mango-fallback rule above applies only to with-skip and with-limit. It does NOT apply to with-count. with-count reads CouchDB view metadata and never falls back to Mango.

Index selection algorithm

Prefix test: an index qualifies as a candidate if its :fields vector starts with the bound args.N strings in order.

Candidate ranking: among all qualifying candidates, the engine selects the one with the shortest field list (closest match to the bound positions).

Defterm PK priority: the PK-exact-match path (Strategy 1) is checked before the write-index paths.

What callers must do

Ensure the bound args in the clause call match a declared view index. If the args don't match, the runtime falls back to Mango and skip/limit become no-ops.

For example, FolderIndex indexes on args.1. A clause call that binds args.1 to a folder ID will resolve through FolderIndex and skip/limit will work. A clause call that only binds args.2 won't match FolderIndex, will fall back to Mango, and skip/limit will be ignored.

The mango-query built-in

mango-query is an explicit built-in term for issuing a raw CouchDB Mango query against a datastore. It is distinct from the automatic Mango fallback that fires in Strategy 3 — it does not go through the index-selection algorithm at all. Its signature is:

(mango-query Ds Mango Object)
  • Ds — a bound datastore ID.
  • Mango — a bound map containing the Mango selector (e.g. {"selector" {"field" {"$eq" Value}}}).
  • Object — output variable; bound to each matching document map.

Mandatory enable-full-scan bookends

Every use of mango-query requires explicit (enable-full-scan) / (disable-full-scan) bookends — no exceptions.

The engine reads :omega-db.run/allow-full-scan? from the OQL execution context before issuing the Mango call. If the flag is not set (i.e. enable-full-scan has not been called), find-docs throws omega/query/full-scan immediately — the query never executes.

This is different from Strategy 3. In Strategy 3, the full-scan flag gates whether the automatic Mango fallback is allowed when no write-index matches. With mango-query, the Mango call is always deliberate — but the flag is still checked, so the requirement is unconditional.

Correct pattern:

(enable-full-scan)
(if (mango-query Ds Mango Object)
  (= Result {"found" true "data" Object})
  (= Result {"found" false}))
(disable-full-scan)

The bookend pair must wrap every invocation site. If mango-query is called inside a captured helper, the bookends must be placed in the helper's body, not in the caller — each execution context has its own ctx.

Common mistake: placing the bookends only around the outer clause body but calling mango-query inside a nested helper. The helper runs in its own sub-context and does not inherit the flag.

Singleton-row term lookup

Some terms hold exactly one row by design — a global config, a singleton service descriptor. The natural call shape for reading that row is arity-1 with the single arg as the output:

(Qo.Schema.Schema/config-data Config)

There is no input arg to constrain on. This shape sits at the seam of the engine's index-matching model and has no clean answer in OQL today. The empty bound-arg set matches no declared index — no view-key prefix can be constructed, so no view query is possible. Mango is the only fallback, which fires the omega/query/full-scan throw.

The only working answer today is to bookend the call with (enable-full-scan) / (disable-full-scan), with a comment pointing back to this pattern. A defterm doesn't help because a defterm with an empty primary key is not a documented or supported declaration. This is a placeholder for a missing engine pattern — not an over-binding fix that was deferred.

Adjacent shapes that are NOT the singleton case:

  • By-id lookup of a known row — if the caller has the row's id or any indexed field, bind it and read through the existing index.
  • Per-parent singleton — a term that holds one row per parent is a normal pkey-on-parent-id lookup.
  • Cardinality-1 by accident — a term that happens to have one row in a particular environment but is not architecturally singular has a normal index.

OQL execution model overview

Every term takes a solution set in, transforms it, and produces a new solution set out. Terms are not expressions — they do not return values that can be nested inside map literals or other terms.

;; WRONG — the runtime does not evaluate (if ...) inside the map literal
(= Result {"key" (if (= X "") "a" "b")})

;; CORRECT — bind first, reference later
(if (= X "") (= Status "a") (= Status "b"))
(= Result {"key" Status})

Think of each term as a CTE (Common Table Expression) in SQL: the engine takes the current solution set, evaluates the term for each row, and joins the results back in. This is why OQL can blow up: 1000 rows from a page query joined with a prop-vals lookup is a 1000-way join.

Index matching rules:

  • Check the defterm's primary key and indexes before writing any query against it.
  • Bind ONLY the arguments that match an index — use _ for everything else.
  • Row-index keys are always arrays, even for single-column primary keys: ["value"] not "value".
;; Row index lookup — key must be an array
(Qo.Db.Prop/row-index FolderId ["https://example.com"] PageId)   ;; CORRECT
(Qo.Db.Prop/row-index FolderId "https://example.com" PageId)     ;; WRONG

No transactions: datastore writes happen immediately during execution. If a term writes data and a later term fails, the write is not undone.

Deprecated: def-query / with-query. Older code uses (def-query Q [...] body) + (with-query Q) for scoping. Do not use these in new code — they cause silent no-return failures when used in stored implementations invoked via dispatch. Use with-table-if for conditionals and inline (:- ...) clauses for scoping.
