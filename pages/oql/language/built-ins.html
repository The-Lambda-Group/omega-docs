Primitives that look like they should exist but don't

OQL's built-in inventory is smaller than Clojure's or Common Lisp's. Several primitives that agents reach for by training default are not implemented.

Name agents reach forStatusCanonical workaround
string-lengthDoes not exist.No primitive returns string length. Inspect a string by binding it directly into Result and letting the caller read it.
string-take / string-substr / substringNone exist.The string toolkit is exactly: string-split, string-concat (binary), string-contains, string-replace, re-find, json-stringify. Slicing by index is not supported.
keysDoes not exist.Use the K-binding-then-fold idiom: (get Map K _) (fold [] K append KeysList).
assoc (as a fold reducer)Does not exist.When folding into a map, use append with key-value pair entries: (= Entry [El El]) (fold {} Entry append Set).
null / nilNeither exists as a literal.OQL has no null literal. For an empty-map sentinel use (= Result {}). For a scalar "no value" sentinel use a string such as (= Result "none"). Writing (= Result null) fails at runtime with no clear error message. For nullable gets, use if (always produces exactly one solution regardless of condition cardinality) rather than with-table-if.

Core

(= X 5)                              ;; unify X with 5
(defined X)                          ;; check if symbol is bound
(return [Result])                     ;; return from a query
(uuid Id)                             ;; generate UUID
(time-now Timestamp)                  ;; current ISO timestamp
(throw "TAG" DataMap)                 ;; raise a typed OQL exception

throw — raising a typed exception

Signature: (throw Tag DataMap) — exactly two arguments, both required.

ArgDirectionTypeDescription
Taginputstring literalIdentifies the error type. Must be a string literal, not a variable.
DataMapinputmapCarries error context. Must be a pre-bound symbol — never a map literal written inline at the call site.
;; WRONG — throws at runtime with a cryptic error
(throw "CAMPAIGN_NOT_FOUND" {"campaign-id" CampaignId "status" Status})

;; RIGHT — bind the map first, then pass the symbol
(= ErrData {"campaign-id" CampaignId "status" Status})
(throw "CAMPAIGN_NOT_FOUND" ErrData)

rand — random double

(rand Output)                        ;; uniformly random double in [0.0, 1.0)

Range: [0.0, 1.0). Output type: double. Not seedable, not cryptographic, never throws. In a multi-solution query, each solution gets its own independent call to rand.

rand-int — random integer in a range

(rand-int Min Max Output)            ;; uniformly random integer in [Min, Max)

rand-int validates its inputs and throws a typed OQL exception on violations:

ConditionException type
Min or Max is unboundRAND_INT_UNBOUND_ARG
Min or Max is not an integerRAND_INT_NON_INTEGER
Min >= MaxRAND_INT_BAD_RANGE

Map Operations

(get Map "key" Value)                 ;; read a key (unifies Value)
(get-in Map ["a" "b"] Value)          ;; nested read (unifies Value)
(set Map "key" NewVal Result)         ;; set a key (returns new map)
(merge Map1 Map2 Result)             ;; merge two maps (Map2 wins on conflict)
(select-keys Map ["k1" "k2"] Result) ;; return new map with only the listed keys
(zip Keys Values Map)                ;; create a map from parallel key/value arrays

merge: when both maps have the same key, Map2's value wins.

zip creates a map from two parallel arrays and also works in reverse — given keys and a map, extracts the values:

;; Forward: keys + values -> map
(zip ["name" "age"] ["Alice" 30] Result)   ;; Result = {"name" "Alice" "age" 30}

;; Reverse: keys + map -> values
(zip ["name" "age"] Values {"name" "Alice" "age" 30 "city" "NYC"})
;; Values = ["Alice" 30]

Array / List Operations

(get-list Map ["k1" "k2" "k3"] Values)   ;; extract values for keys as an array
(length Array Len)                         ;; array length
(append Array Item Result)                 ;; append item to array
(concat Array1 Array2 Result)              ;; concatenate two arrays
(partition Array N Chunks)                 ;; split array into chunks of size N

String Operations

(string-split Path "/" Segments)          ;; split string
(string-concat "Hello " Name Greeting)   ;; concatenate strings (strictly binary)
(string-contains Str "sub" Result)        ;; check substring
(string-replace Str "," "" Clean)         ;; replace substring
(re-find Str "\\d+" Match)               ;; regex find
(json-stringify Obj Str)                  ;; JSON: object <-> string (bidirectional)

string-concat is binary, not variadic. It always takes exactly two input strings and an output. To concatenate three or more strings, chain calls:

(string-concat "https://api.example.com/contacts/?locationId=" LocationId Url0)
(string-concat Url0 "&limit=100" Url)

json-stringify is bidirectional — there is no json-parse

Direction is determined by which argument is bound:

  • Stringify (object to string): bind the first arg, leave the second unbound.
  • Parse (string to object): leave the first arg unbound, bind the second.
;; Stringify
(= Data {"name" "Alice" "age" 30})
(json-stringify Data JsonStr)

;; Parse
(get Res "body" RawBody)
(json-stringify Parsed RawBody)   ;; Parsed is now a map you can (get ...) into

Calling (json-parse Body Result) produces a 500 with no hint that you need json-stringify with reversed binding.

Probing validity: use a named symbol, not _. (json-stringify _ Text) always fails. Use a named symbol instead:

;; WRONG — always fails
(when-not (json-stringify _ Text)
  (throw "PARSE_ERROR" ParseErr))

;; RIGHT
(when-not (json-stringify ParseProbe Text)
  (throw "PARSE_ERROR" ParseErr))

json-stringify parse direction is strict — rejects raw control characters. A json-stringify parse-direction call fails with omega/jvm/Exception: JSON error (missing entry in object) when the input string contains raw control characters (ASCII 0–31 other than space, newline, tab) that are illegal in JSON. The fix is to fix the producer — tighten the LLM prompt so the model emits clean output.

json-stringify with integer values — direction trap

When stringifying integers extracted from JSON (e.g. position, delay_days), the integer is an OQL Long, not a string. The stringify direction requires the Long as the first argument:

;; CORRECT — stringify direction: object first, string second
(json-stringify IntegerVar StringVar)   ;; IntegerVar = 1 → StringVar = "1"

;; WRONG — parse direction with a Long in second position
(json-stringify StringVar IntegerVar)   ;; throws ClassCastException: Long cannot be cast to String

The error ClassCastException: java.lang.Long cannot be cast to java.lang.String with no stack hint means the direction is inverted. Check every json-stringify call that touches a number.

Also: (fold [] _ append _) is invalid — _ is a wildcard and cannot be folded over. Name the variable: (run-term Helper ... ItemDone) then (fold [] ItemDone append _). Using _ throws IllegalArgumentException: Don't know how to create ISeq from: clojure.lang.Symbol.

HTTP

http-request is the general-purpose HTTP term and the only safe HTTP primitive.

;; GET with query params
(= Req {"method" "get"
         "url" "https://api.example.com/contacts"
         "headers" {"Authorization" "Bearer token123"}
         "query-params" {"locationId" "abc" "limit" "100"}})
(http-request Req Res)
(get Res "status" Status)    ;; 200
(get Res "body" Body)        ;; response body as string — must be parsed separately

;; POST with JSON body
(= Req {"method" "post"
         "url" "https://api.example.com/contacts"
         "headers" {"Authorization" "Bearer token123"
                    "Content-Type" "application/json"}
         "body" "{\"firstName\": \"Jane\"}"})
(http-request Req Res)

The body field is always a raw JSON string, never a parsed map. Parse it with json-stringify in parse-direction before accessing fields. Always branch on status BEFORE parsing body.

;; Full error-handling pattern
(= Req {"method" "post" "url" Url "headers" Headers "body" JsonBody})
(http-request Req Res)
(get Res "status" Status)
(when-not (= Status 200)
  (get Res "body" ErrBody)
  (= ErrData {"status" Status "body" ErrBody "url" Url})
  (throw "HTTP_ERROR" ErrData))
;; only parse body after confirming success
(get Res "body" RawBody)
(json-stringify ParsedBody RawBody)
(get ParsedBody "id" ResultId)

Deprecated method-specific HTTP terms

http-get-request, http-post-request, and http-delete-request are deprecated. They do not capture HTTP error codes. Any clause using them that hits a non-2xx response will silently fail to bind the expected response fields. Use http-request with the "method" key instead.

Symbols in Literal Arguments

Symbol bindings do not resolve inside literal map or list arguments passed directly to any term. The contents of the literal are read at parse time, before the runtime has bound symbols — so {"key" MySymbol} becomes a literal map with the text "MySymbol", not the value of MySymbol.

The fix is always the same: bind the map to a variable with (=) first, then pass the variable.

;; WRONG — Payload stays as a literal symbol name
(throw "error-tag" {"payload" Payload "count" 42})

;; RIGHT — bind the map to a symbol first, then pass the symbol
(= Data {"payload" Payload "count" 42})
(throw "error-tag" Data)

Same pattern applies to run-page, set-data, and any term that accepts a map or list argument. This also applies to inline key vectors passed to index terms — bind the key list first:

;; WRONG — Tid stays symbolic inside the inline list
(Qo.Public.OqlApi.Db.Prop/row-index StrategiesDbId [Tid] RowPageId)

;; RIGHT — bind the key vector first
(= RowKey [Tid])
(Qo.Public.OqlApi.Db.Prop/row-index StrategiesDbId RowKey RowPageId)

List-wrapper edge case

A bound map can still leak through as a symbolic placeholder when wrapped in an inline one-element list literal. Use fold + append to build a list from maps instead:

;; Safe pattern — build the list through fold
(= NextArg {"step" "Load Subjects" "skip" NextSkipStr "limit" BatchSize})
(fold [] NextArg append NextArgs)
(= Result {"has-more" "true" "next-args" NextArgs})

Data Transformation Patterns

Preparing data for write-table

write-table expects columnar input: {"file-id" FolderId "data" {"header" [...] "rows" [...]}}. The efficient pattern:

  1. Define the header array once (this is the column order contract).
  2. For each record, use get-list or zip (reverse) to extract values in that exact order.
  3. Pass {"header" Header "rows" [Values]} to write-table.
(= Header ["name" "email" "status"])
(get-list Record ["name" "email" "status"] Row)
(= Input {"file-id" PageId "data" {"header" Header "rows" [Row]}})
