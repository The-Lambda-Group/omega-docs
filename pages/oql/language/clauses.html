What a clause is

A clause defines a rule — when its head is true based on its body. Clauses are defined with (:- ...):

(:- (Head Arg1 Arg2 Result)
    (some-term Arg1 Intermediate)
    (other-term Intermediate Result))

Read this as: Head is true if the body terms are all true.

Clause types

Stored clauses

When a clause head is namespaced with a datastore and ends with !, it is stored in that datastore and persists across queries:

(datastore Qo.Page "omega/query-omega/page")

(:- (Qo.Page/visible! PageId)
    (Qo.Page/page-object _ ParentId PageId _)
    (Qo.Page.Bl/page-block _ ParentId _ Block)
    (get Block "page-id" PageId))

The ! suffix tells the engine to write the clause to the datastore. Without !, calling the clause reads from the datastore. This is the read/write convention:

(Qo.Page/visible PageId)   ;; READ — query the stored clause
(Qo.Page/visible! PageId)  ;; WRITE — define/store the clause

In-memory clauses

When a clause head has no namespace prefix and no !, it exists only as a symbol binding in the current query. Think of it like a local function:

(:- (ParseSales RawSales Sales)
    (with-table-if (= RawSales "")
      [RawSales Sales]
      (= Sales 0)
      (json-stringify Sales RawSales)))

In-memory clauses are used for helper logic inside an implementation, implementation clauses registered with set-implementation-clauses, and captured clauses passed into stored clauses via the capture mechanism.

Inner clauses (experimental)

You can define an in-memory clause inside another clause's body. It is scoped to that execution. Warning: inner clauses are highly experimental. Stored clause writes inside inner clauses do not persist. Only use inner in-memory clauses for helper logic within a single query execution.

Clause size and decomposition

A clause body is a unit of debugging. When something throws inside a body, the stack trace dumps the entire body — the larger the body, the harder it is to localize the failure.

The discipline:

  1. 5–10 lines per clause body. If a body grows past ~10 terms, split it into smaller helper clauses.
  2. No nested with-table-ifs in any clause body. Use flat sequential with-table-ifs instead — each one does one piece of classification, the next reads the result via normal per-row unification.
  3. Compose via (call SmallHelper ...), not via inline (and ...) blocks. A 90-line (and ...) holding 15 distinct concerns is the anti-pattern. Stack traces from inside the (and ...) lose the branch boundary.
  4. Each helper that calls captured helpers must declare its own {"capture" [...]}. Captures do not propagate through nested call / run-term.

Decomposition shape

The canonical shape for a workflow OnEvent or other large entry-point clause:

  • One BuildCtx/ResolveCtx helper that resolves the page hierarchy and database ids into a single Ctx map.
  • A handful of read helpers (one per source) each 5–10 lines.
  • One or more compute helpers (batch, build user message, dispatch LLM, etc.) each 5–10 lines.
  • Write helpers (one per destination), each invoked via (call WriteHelper ...).
  • A thin entry-point clause (OnEvent, Run, Install, etc.) — usually 5–15 lines — that orchestrates the sequence of (call ...) invocations.

Call graph shape

Every impl has a call graph — the directed tree of (call X ...) invocations rooted at OnEvent. The rule: 2–4 children per non-leaf clause.

  • Leaf clauses do real work: one scan, one write, one string-concat chain. They have no (call ...) invocations.
  • Non-leaf clauses orchestrate: their body is mostly (call Child1 ...), (call Child2 ...), etc. Keep to 2–4 calls.
  • Top-level OnEvent orchestrates 2–3 phase-level orchestrators, not 10+ leaves directly.

The flat anti-pattern: an OnEvent that directly calls 12 sequential helpers is a flat orchestrator. A runtime failure anywhere in that body is hard to bisect. Restructure into 2–3 phase orchestrators, then verify each phase independently.

Implementation clause arity

OQL always dispatches implementation (protocol method) clauses with exactly 2 positional args: Exec and Result. The clause signature is always:

(:- (MyClause Exec Result)
    ...)

Extra method arguments are not passed as additional positional params. They arrive inside Exec under ["args" 0 ...] and must be extracted with get-in:

(:- (InstallBranch Exec Result)
    (get-in Exec ["args" 0 "branch-name"] BranchName)
    (get-in Exec ["args" 0 "library-id"] LibraryId)
    (= Result {"status" "ok"}))

Writing a 3-arg signature like (:- (InstallBranch Exec Input Result) ...) is wrong — the dispatcher always calls with 2 args, so the clause will never unify.

Closures / Capture

Clauses can capture symbols from the outer scope using the capture syntax:

(:- (MyClause A B Result {"capture" [SomeOuterVar]})
    (+ A SomeOuterVar Result))

The capture map must be the LAST element of the clause head tuple. Placing {"capture" [...]} anywhere in the body is silently ignored — no error is raised, captures simply do not take effect. The runtime symptom is count not supported on this type: Symbol.

;; WRONG — capture map is the first body term; it is silently ignored
(:- (OnEvent Exec Result)
    {"capture" [ResolveCtx WriteResults]}
    (call ResolveCtx Exec Ctx)
    (call WriteResults Ctx Result))

;; CORRECT — capture map is the last element of the head tuple
(:- (OnEvent Exec Result {"capture" [ResolveCtx WriteResults]})
    (call ResolveCtx Exec Ctx)
    (call WriteResults Ctx Result))

Captures are not transitive

Capture scope is per-clause only. When OnEvent captures [ProcessAnalyst] and calls (call ProcessAnalyst ...), ProcessAnalyst runs inside its own scope. If ProcessAnalyst needs ThinkPhase, it must declare {"capture" [ThinkPhase]} itself.

The rule: every (call X ...) must have X in that clause's own {"capture" [...]}. There is no capture inheritance across call boundaries.

Symptom of a broken capture chain:

TermExecutionRunnable found for class: clojure.lang.Symbol
;; WRONG — two-level chain where OnEvent hopes ProcessAnalyst's captures cascade up
(:- (OnEvent Exec Result {"capture" [ProcessAnalyst]})
    (call ProcessAnalyst Exec Result))

(:- (ProcessAnalyst Exec Result {"capture" [ThinkPhase FormatPhase]})
    (call ThinkPhase Exec ThinkOut)
    (call FormatPhase ThinkOut Result))

;; CORRECT — OnEvent flattens: captures every helper it will ever call, directly
(:- (OnEvent Exec Result
             {"capture" [ResolveAnalystCtx ProcessThinkPhase ProcessFormatPhase]})
    (call ResolveAnalystCtx Exec Ctx)
    (call ProcessThinkPhase Ctx ThinkOut)
    (call ProcessFormatPhase ThinkOut Result))

Captured in-memory helpers are only callable from the clause that declared them

A captured in-memory helper is callable only from the clause whose head declares the {"capture" [...]} — not from any clause invoked via (call ...) from that declaring clause. The fix is always to flatten: the clause that has access to captured helpers must call them directly.

Related scoping gotchas

  • Lexical scope — what survives into a clause body, the push-time vs runtime scope split, and the "Capture Only Works With call" rule.
  • with-table-if header must include captured clause symbols — even after capture has threaded a clause symbol into the surrounding body, a with-table-if branch needs that symbol listed in its header to see it.
  • run-term does not work on in-memory clauses — capture targets must be stored functors when reached via run-term. For in-memory captured helpers, use call. The only valid run-term shape for captured in-memory helpers is the partition-before-fold pattern or per-row iteration at scale (K ≥ ~30 rows).

Captured helpers must be defined before they are captured

Symptom: a stored implementation pushes successfully, but qo run fails at runtime with:

omega/jvm/IllegalArgumentException
No implementation of method: :run-with-args ... found for class: clojure.lang.Symbol

The root cause: a captured helper was referenced in a clause's {"capture" [...]} before it was defined in the same push file. In OQL, the top-level scope of an implementation file runs once at push time — earlier statements cannot refer to symbols that are defined later. Define all helper clauses before the clause that captures them.

fold in clause bodies

fold is a solution-wide reducer. Two edge-case behaviors can surprise you in clause bodies:

fold with zero input solutions produces zero output solutions

fold does not bind its accumulator to the Init value when the upstream has zero solutions. If the upstream term produces zero rows, fold produces zero rows — the Agg symbol is left unbound. Use the count-then-fold workaround documented in the Reducers reference.

The count-then-fold workaround

(with-table-if (<upstream scan producing PerRowSym>)
  [<header including List>]
  (fold [] PerRowSym append List)
  (= List []))
;; List is bound on both paths — fold path and empty path.
(length List Count)  ;; outside the with-table-if; works on both paths

The fold symbol must be per-row-unique — one distinct value per upstream row — because fold deduplicates over its input symbol.

Hard Rules

These are non-negotiable rules for writing OQL. Every one of them has produced real data loss, index corruption, or silent wrong-answer bugs in production. Read these before writing any line of OQL.

  1. Never open a system/ datastore directly. The system/ namespace holds subscriptions and internal runtime state. Direct access can wipe critical state the runtime depends on. Use a public API clause instead. (datastore X "system/...") is forbidden.
  2. Never open internal data-layer datastores directly. The internal data layer lives under omega/query-omega/data/.... Opening it directly bypasses access control, schema guarantees, and every invariant the public API enforces. Use Qo.Db.Prop/, Qo.Page/, or Qo.Public.OqlApi.*/ instead. For read-only row counts, use omega-cli db-name + curl GET /<db>/ to read doc_count.
  3. Never run mutations in scratch files. Scratch files are read-only. No delete!, no refresh, no !-suffixed write terms. Mutations belong in deployed implementations invoked via qo push + qo run. Using (Ds/term! Args) in a scratch query invokes the write pathway and throws WRITING_SYMBOLS if Args contains unbound symbols.
  4. Never use refresh on any datastore. refresh wipes the datastore and breaks the CouchDB indexes that back it. The indexes do not rebuild automatically. There is no legitimate workflow that calls refresh.
  5. Never use boolean true / false in OQL — use strings. OQL's solution engine handles native booleans incorrectly, producing silent wrong answers. Always use the strings "true" and "false". This applies everywhere: term arguments, map values, comparisons, defaults. (= Flag true) is forbidden. Use (= Flag "true").

Treat a temptation to violate any of these as a sign that you are solving the wrong problem.

Semi-symbolic map arguments

A semi-symbolic map is a map literal whose values include unbound symbols at the call site — for example {"key" X} where X is bound only at runtime. Support for semi-symbolic maps varies by call context.

In-memory clause calls via call — fully supported

Passing a semi-symbolic map directly to call works when the symbol is bound at call time:

(call Clause {"key" X} Out)

The workaround of pre-binding the map — (= M {"key" X}) (call Clause M Out) — is still valid but is no longer required.

Stored clause calls where the body uses get — supported

Inside a stored clause body, the get built-in resolves symbol values out of map literals. If the body receives a map argument and extracts fields with get, semi-symbolic maps passed in from the caller work correctly because get performs the resolution at runtime against the current bindings.

Defterm fact terms — not supported for index selection

Defterm fact terms (e.g. T/item) require fully-resolved scalar arguments for index selection. Passing an unbound symbol as a key field fails with a no-matching-index error:

;; FAILS — PageId is unbound; the index cannot be selected
(T/item {"id" PageId} Out)

;; CORRECT — bind the scalar first, then call
(= PageId "some-page-id")
(T/item {"id" PageId} Out)

This is intended behavior. The engine must pick an index before executing the scan, which requires concrete scalar values for every indexed field.
