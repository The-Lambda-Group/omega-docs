Control Flow

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

OQL's control flow operates on solution tables. Understanding the distinction between universal quantifiers (when, when-not, if) and per-solution branching (with-table-if) is critical. Read this entire document before writing any conditionals.

when / when-not / if (universal quantifiers)

These operate as "if any" — they pass or fail based on whether any solution in the current solution set satisfies the condition. They do not branch per-solution.

;; (when) — runs body if the condition succeeds for any solution
(when (= Step "score")
  (= PageName "Score Agents"))

;; (when-not) — runs body if the condition fails for all solutions
(when-not (get Data "skip" Skip)
  (= Skip 0))

;; (if) — binds one of two values based on whether condition succeeds
(if (get Data "continue" "false")
  (= ShouldContinue "false")
  (= ShouldContinue "true"))

Use when for flat dispatch (like a switch/case):

(when (= Step "score")  (= PageName "Score Agents"))
(when (= Step "filter") (= PageName "Filter Agents"))
(when (= Step "batch")  (= PageName "GHL Sync"))

when-not in multi-solution context is all-or-nothing

(when-not condition) is a universal quantifier: it succeeds only when the condition produces zero solutions across the entire current solution table. In a multi-solution context (e.g. after prop-vals-by-folder returns N rows), when-not is not a per-row filter — it passes through all solutions only if no solution in the table satisfies the condition.

;; WRONG — filters to only rows where NO solution in the table has a block
(prop-vals-by-folder FolderId RowPid RowPv)
(page-by-id RowPid RowPage)
(when-not (get-content RowPage "Content" _))   ;; passes ALL or NONE

To filter per-row, use if for per-solution branching, or restructure to avoid the need (e.g. collect all rows unconditionally and rely on idempotent writes):

;; Idempotent write approach — process all rows, overwrite is safe
(prop-vals-by-folder FolderId RowPid RowPv)
(get-in RowPv ["prop-vals" "content"] Content)
(= Pair [RowPid Content])
(fold [] Pair append Pairs)
;; pass Pairs to a helper that writes unconditionally

with-table-if (per-solution branching)

with-table-if branches per solution. The condition evaluates within the parent solution, and each branch creates its own table solution.

(with-table-if Condition
  [Var1 Var2 Result]          ;; table head — variables in scope for both branches
  ThenBranch                   ;; runs for solutions where Condition succeeds
  ElseBranch)                  ;; runs for solutions where Condition fails

SQL analogy

Think of with-table-if as four steps: a SELECT DISTINCT <schema> projection, then an INNER JOIN for the then-branch, a LEFT OUTER JOIN for the else-branch, and a UNION of both halves. Three key consequences:

  1. The SELECT DISTINCT at step 1 is where row collapse happens — rows with identical values for every schema variable merge into one.
  2. The then-branch is an inner join — (= true false) fails unification, which drops the row.
  3. The else-branch is conventionally a left outer join — almost always (= true true), a no-op pass-through.

The existence-check fallacy (then-branch runs K times, not once)

This is one of the most common with-table-if mistakes, and it causes exponential output blowup that is hard to diagnose.

The mental model: "for each solution from the condition, run then-branch; if the condition has zero solutions, run else-branch once."

This is NOT an existence check. If prop-vals-by-folder generates K rows, every downstream term in the then-branch runs K times. If your then-branch calls a helper that generates M solutions, you get K×M solutions — a hidden cross-product.

;; BAD — "existence check" with multi-row condition
;; If MemoryFolder has K rows, the then-branch runs K times
(with-table-if (prop-vals-by-folder MemoryDbId MemFolderPid _)
  [MemoryDbId MemFolderPid LatestRev]
  (run-term ReadLatestRevision MemoryDbId LatestRev)   ;; runs K times
  (= LatestRev "0000"))

;; GOOD — aggregate the condition first; then-branch runs exactly once
(with-table-if (prop-vals-by-folder MemoryDbId MemFolderPid _)
  [MemoryDbId MemFolderPid MemRows]
  (fold [] MemPid append MemRows)
  (= MemRows []))

(with-table-if (> (length MemRows 0) 0)
  [MemoryDbId MemRows LatestRev]
  (run-term ReadLatestRevision MemoryDbId LatestRev)   ;; runs exactly once
  (= LatestRev "0000"))

To get true existence-check semantics: aggregate the condition via fold first, or use run-page-by-id against a query page that returns at most one result.

Preserving per-row identity

This is the most common source of silent row loss.

When the solution table has multiple rows, the merge after branching is effectively a SELECT DISTINCT <schema> union. If two rows have identical values for every variable in the schema, they collapse into one row.

Always include at least one variable that is unique per row in the schema — typically a primary key, URL, or UUID.

;; BAD — silent row collapse if two agents have the same AgentStatus and HttpStatus
(with-table-if (and (= AgentStatus "create")
                    (= HttpStatus 422))
  [AgentStatus HttpStatus]     ;; schema has no per-row identity
  (= true false)
  (= true true))

;; GOOD — AgentLink anchors per-row identity
(with-table-if (and (= AgentStatus "create")
                    (= HttpStatus 422))
  [AgentLink AgentStatus HttpStatus]   ;; AgentLink is unique per agent
  (= true false)
  (= true true))

Dropping rows from the solution

A with-table-if can be used to drop rows by having the then-branch fail to unify. The idiomatic form is (= true false):

(with-table-if (and (= AgentStatus "create")
                    (= HttpStatus 422))
  [AgentLink AgentStatus HttpStatus]
  (= true false)    ;; then: fail the branch — row drops from the solution
  (= true true))    ;; else: pass through unchanged

Throw in the else-branch does not halt the enclosing clause

A throw inside a with-table-if branch does NOT propagate out of the branch and stop the enclosing clause body. The throw fires within the branch's own scope and terminates that solution path, but OQL continues executing subsequent terms in the enclosing clause with whatever bindings survived.

The critical consequence: any schema variable that was supposed to be bound by the then-branch remains unbound as a raw clojure.lang.Symbol.

Fix: move any term that depends on schema variables inside the then-branch, not after the with-table-if.

;; BAD — (call Func ...) runs even when Func was never bound
(with-table-if (lookup-deletable-functor DbId Name Func)
  [DbId Name Func Deletable Result]
  (= Deletable {"id" Name})
  (throw "NOT_FOUND" {"name" Name}))
(call Func Deletable Result)   ;; runs even when Func is unbound

;; GOOD — move the call inside the then-branch
(with-table-if (lookup-deletable-functor DbId Name Func)
  [DbId Name Func Deletable Result]
  (and (= Deletable {"id" Name})
       (call Func Deletable Result))
  (throw "NOT_FOUND" {"name" Name}))

When the condition is batch-wide, not per-row

(defined X) checks whether the symbol X is bound in the current lexical context, not whether a specific row has a value. Inside a branch where X is in scope, (defined X) is always true — every row sees a bound X.

Never use (defined X) alone as the condition for with-table-if. Always combine it with a concrete per-row predicate.

;; BAD — (defined X) is effectively universal
(with-table-if (defined SuccessId)
  [AgentLink SuccessId GhlContactId AgentStatus]
  (and (= GhlContactId SuccessId) (= AgentStatus "created"))
  (= true true))

;; GOOD — combine with a per-row value check
(with-table-if (and (= AgentStatus "create") (defined SuccessId))
  [AgentLink SuccessId GhlContactId AgentStatus]
  (and (= GhlContactId SuccessId) (= AgentStatus "created"))
  (= true true))

Body scope rules

Inside a with-table-if branch body:

  1. Only symbols listed in the schema [A B Result] flow in from the outer scope.
  2. Symbols bound in CONDITION do NOT automatically flow into the branch body — include them in the schema OR re-bind inside the body (the check-then-rebind idiom).
  3. Symbols bound inside the branch body are local to that branch and do not escape to downstream clauses.
;; BAD — Val is bound by the condition but not listed in the schema
(with-table-if (get Data "key" Val)
  [Data Result]
  (= Result {"found" Val})       ;; Val is unbound here
  (= Result {"found" "default"}))

;; GOOD — list Val in the schema so it flows into the body
(with-table-if (get Data "key" Val)
  [Data Val Result]
  (= Result {"found" Val})
  (= Result {"found" "default"}))

The Ctx-threading pattern

Large implementations typically roll configuration into a single Ctx map at the top of the query and thread Ctx through every with-table-if schema. Inside each branch body, pull the fields you need back out of Ctx:

(= Ctx {"filtered-folder-id" FilteredFolderId
        "ghl-contacts-folder-id" GhlContactsFolderId
        "auth-header" AuthHeader})

(with-table-if (and (= AgentStatus "create") ...)
  [AgentLink Ctx AgentStatus Res HttpStatus]
  (and (get Ctx "auth-header" FlowAuthHeader)
       (get Ctx "ghl-contacts-folder-id" FlowGhlFolderId)
       ...)
  (= true true))

The binding list as JOIN key

The binding list is not just a visibility declaration — it is the JOIN key between the outer solution table and the inner branch. Any outer-scope variable the inner branch needs must appear in the binding list. If you omit a variable, the inner branch runs independently and results lose their association with the outer rows.

;; WRONG — Data is not in the binding list; association is severed
(with-table-if (get Data "campaign_id" SpecificId)
  [SpecificId CampaignIds]
  (= CampaignIds [SpecificId])
  ...)

;; RIGHT — Data is in the binding list; association preserved
(with-table-if (get Data "campaign_id" SpecificId)
  [Data SpecificId CampaignIds]
  (= CampaignIds [SpecificId])
  ...)

Nesting

Nesting is allowed for multi-step parsing but adds real costs: stack traces lose the branch boundary, and schema discipline compounds. Prefer flat sequential with-table-ifs over nesting when possible.

;; GOOD — flat, sequential. Each step does one thing.
(with-table-if (= RawSales "")
  [AgentLink RawSales SalesBucket]
  (= SalesBucket "empty")
  (= SalesBucket "nonempty"))

(with-table-if (and (= SalesBucket "nonempty") (number RawSales))
  [AgentLink RawSales SalesBucket Sales]
  (= Sales RawSales)
  (= true true))

There is no not operator

OQL does not have a (not ...) operator. The correct way to express "complement of CONDITION" is the else-branch of a single with-table-if:

;; BAD — (not ...) is not an OQL primitive
(with-table-if (or (not (get [200 201] _ HttpStatus))
                   (not (get ParsedRes "customFields" _)))
  [AgentLink Result]
  (= Result {"status" "error"})
  (= true true))

;; GOOD — a single with-table-if with both branches
(with-table-if (and (get [200 201] _ HttpStatus)
                    (get ParsedRes "customFields" _))
  [AgentLink Result]
  (= Result {"status" "ok"})
  (= Result {"status" "error" "reason" "missing-fields"}))

whenFlat dispatch — run body if condition matches any solution
