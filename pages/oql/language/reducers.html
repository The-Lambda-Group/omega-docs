Reducers

Visibility

Owner: omega | Workspace: OmegaAI | Visibility: Public

These operations work on the solution table — they aggregate, filter, or reorder rows across the solution set.

(fold Init Sym Func Agg)                  ;; reduce across solutions
(with-count Count Term)                   ;; count solutions matching a term
(with-order-by [Sym] Body)               ;; order solutions by symbol value
(with-group-by [Sym] Body)               ;; group solutions by symbol value
(with-unique-by [Sym] Body)              ;; deduplicate solutions by symbol value
(with-limit N Body)                       ;; limit to first N solutions
(with-skip N Body)                        ;; skip first N solutions
(with-sort Dir Body)                      ;; sort direction: "asc" or "desc"

fold

Reduces a symbol across all solutions in the table. The function is applied pairwise (like reduce in functional programming):

;; Collect all names into an array
(get Items _ Item)
(get Item "name" Name)
(fold [] Name append Names)              ;; Names = ["Alice", "Bob", "Carol"]

;; Sum a numeric value
(get Items _ Item)
(get Item "score" Score)
(fold 0 Score + Total)                   ;; Total = sum of all scores

fold takes four arguments: Init (initial accumulator), Sym (symbol to reduce over), Func (reducer function), Agg (output symbol).

Reducer functions are language primitives, not first-class values. append, prepend, +, *, and any other reducer are literal terms in the OQL grammar — they cannot be stored in a symbol. Writing (= Reducer "append") (fold [] Item Reducer List) does not work. If runtime logic determines which reducer to use, write two fully explicit branches selected with with-table-if.

fold pitfalls

fold is solution-wide, not per-row

fold is a solution-wide aggregation — think SQL GROUP BY the whole table. When you write (fold [] P append Fields) inside a with-table-if branch to build one array per row, at batch=1 it works by accident. At batch>1, every row gets the same merged array containing ALL rows' values.

The fix — use append for per-row accumulation:

;; BAD — fold builds ONE shared array across all rows
(with-table-if [ContactId FieldPair] (defined FieldPair)
  (fold [] FieldPair append Fields))
;; batch=2 -> Fields = [pair-A pair-B] (both contacts get BOTH pairs)

;; GOOD — append builds one array per row
(with-table-if [ContactId FieldPair] (defined FieldPair)
  (append Fields FieldPair Fields1))
OperationScopeUse for
foldsolution-widetotals, counts, collecting all distinct values into one result
appendper-rowbuilding an array that belongs to each row independently

fold folds over unique values, not rows

If X is a column whose values are 0 or 1, fold sees two unique values and returns a sum of 1, regardless of how many rows had X = 1. Use with-unique-by to make fold see one value per unique row key:

;; BAD — Sum maxes at 1 even if 500 rows have NeedsCreate
(fold 0 NeedsCreate + Sum)

;; GOOD — Sum = count of rows
(with-unique-by [RowId]
  (fold 0 NeedsCreate + Sum))

Or count rows directly with a literal 1:

(with-table-if [RowId NeedsCreate] (= NeedsCreate "true")
  (fold 0 1 + Count))

fold over zero solutions does not bind its accumulator

fold does not bind its accumulator to the Init value when the upstream has zero solutions. If the upstream has zero solutions, fold produces zero rows — the Agg symbol is left unbound.

;; If no pages match, AllPids is unbound — downstream (return [AllPids]) fails
(Qo.Public.OqlApi.Db.Prop/sec-index FolderId "agent" "researcher" Pid)
(fold [] Pid append AllPids)
(return [AllPids])

The canonical fold-with-default idiom

(with-count Cnt (Qo.Public.OqlApi.Db.Prop/sec-index FolderId "agent" "researcher" _))
(with-table-if (> Cnt 0)
  [FolderId AllPids]
  (and (Qo.Public.OqlApi.Db.Prop/sec-index FolderId "agent" "researcher" Pid)
       (fold [] Pid append AllPids))
  (= AllPids []))
;; AllPids is now bound on every downstream row, regardless of cardinality.

Why this works: with-count always produces exactly one solution — even when the inner term matches nothing — and binds Cnt to 0 in the empty case.

Counting rows in a captured-helper body when with-count is unavailable

When with-count is unavailable (compound predicate, sec-index trailing wildcard, or a captured helper containing prop-vals-by-folder), use the fold-append-per-row-unique-symbol pattern:

(:- (CountRowsMatchingPredicate ScanArgs FilterArgs Count)
    (with-table-if (and (<scan term producing per-row symbols>)
                        (<filter terms>))
      [<header symbols including PerRowUniqueSymbol and RowList>]
      (fold [] PerRowUniqueSymbol append RowList)
      (= RowList []))
    (length RowList Count))

The load-bearing detail: fold must be over a per-row-unique symbol. In a CouchDB-backed prop-vals-by-folder scan, the page-id (Pid) is reliable because each CouchDB document has a unique _id.

Counting source-array length vs fold-accumulator length

Any "are there duplicates in this array?" check must measure the source array's length, not a folded view of it. A uniqueness-validation check that compares post-fold lengths will always report "no duplicates" — both arrays are already deduplicated.

;; RIGHT — measure the input array
(get Drafts _ Draft)
(get Draft "treatment_id" Tid)
(fold [] PairKey append AllKeys)
(length Drafts Total)                ;; measure the input array
(call IntoSet AllKeys UniqueSet)
(fold [] UKey append UniqueKeys)     ;; in IntoSet, fold over unique set keys
(length UniqueKeys UniqueCount)
(with-table-if (= UniqueCount Total) [...] ok throw-duplicate-keys)

Why batch=1 masks fold bugs

With one row in the solution, "all rows" and "this row" are the same set. Both fold pitfalls only surface when the batch contains two or more rows.

with-unique-by

with-unique-by changes what "unique" means for folds inside its body. Instead of deduplicating by the fold symbol, it deduplicates by the specified symbol(s):

;; Problem: ApiCall is 0 or 1 per row. Without with-unique-by,
;; fold sees at most 2 unique values.
(fold 0 ApiCall + ApiCount)              ;; BUG: always 0 or 1

;; Solution: unique-by AgentLink means fold sees one ApiCall per agent
(with-unique-by [AgentLink]
  (fold 0 ApiCall + ApiCount))           ;; CORRECT: sums per agent

with-order-by

Sorts solutions before folding or returning. with-order-by always sorts ascending — there is no direction parameter.

(with-order-by [Score]
  (fold [] Name append SortedNames))

Producing a descending-sorted list

Sort ascending with with-order-by [Key], then fold with prepend instead of append:

;; Descending — highest Score first
(with-order-by [Score]
  (fold [] Name prepend DescNames))

Empirically verified: DescNames is the exact reverse of an ascending fold.

with-group-by

Partitions solutions into groups. Inside the body, fold operates per-group:

(with-group-by [Department]
  (fold 0 Salary + DeptTotal))

with-count

Counts solutions matching a term. Always produces exactly one solution row — including 0 when the term matches nothing. This makes it the building block for empty-handling around fold.

(with-count Total (Qo.Db.Prop/prop-vals FolderId _ _))  ;; Total = row count

All-wildcard form currently NPEs. Passing all wildcards to count every row in a defterm — (with-count Cnt (Qo.Namespace/term _ _ _ _)) — currently fails with omega/jvm/NullPointerException. For a total row count, use omega-cli db-name + curl GET /<db>/ to read doc_count.

Restrict the inner term to a single call. with-count over a compound condition — (with-count Cnt (and Term1 Term2)) — currently fails with a NullPointerException. Wrap the compound logic in an in-memory clause and count over the clause invocation instead.

Four known NPE patterns:

  • sec-index with trailing wildcard
  • compound (and ...) predicate
  • captured helper containing prop-vals-by-folder
  • clause wrapper — any clause that internally delegates to a raw defterm (see below)

For any of these, use the fold-append-per-row-unique-symbol + length pattern documented above.

with-count on clause wrappers vs raw defterm terms

with-count only works on raw defterm terms. Calling it on a clause wrapper that delegates internally to a defterm produces omega/jvm/NullPointerException.

A clause wrapper is a stored clause that calls a raw defterm in its body, often to add filtering, transformation, or a public API surface. For example:

(:- (Qo.Db.Prop/sec-index! ...)
    (Qo.Data.Db.Prop.SecIndex/sec-index ...))

Qo.Db.Prop/sec-index is a clause wrapper. Its body calls Qo.Data.Db.Prop.SecIndex/sec-index, which is the raw defterm. with-count on the wrapper NPEs; with-count on the raw defterm works correctly:

;; BAD — NPEs with omega/jvm/NullPointerException
(with-count N (Qo.Db.Prop/sec-index FolderId "name" "value" _))

;; GOOD — raw defterm works correctly; confirmed with N = 300097
(with-count N (Qo.Data.Db.Prop.SecIndex/sec-index FolderId _ _ _))

How to find the raw defterm for a clause wrapper: search the OQL source for the relevant datastore for (defterm Namespace {"name" "term-name" ...}). The defterm's namespace is the callable term. As a naming convention, raw defterm namespaces often include .Data. in the path (e.g. Qo.Data.Db.Prop.SecIndex) while clause wrappers use a shorter public-API namespace (e.g. Qo.Db.Prop).

This pattern also extends to terms accessed through page-ds. When page-ds returns the actual datastore and the body clause calls the raw defterm directly (e.g. PageDs/page), with-count works correctly — because page-ds resolves to the underlying raw defterm, not a clause wrapper.

Performance: partitions and run-term

fold / with-group-by / with-order-by scale poorly when the solution table contains many distinct group-key values. The idiomatic workaround is to establish the partition boundary before folding: wrap the reducer body in a clause and invoke it once per key via run-term.

Measured: 8.3s with a top-level with-group-by [PageId], 0.5s when the same reducer body was called via run-term per page (21-page pull, 2026-04-24).

There are two valid run-term shapes for captured in-memory helpers:

  1. Partition-before-fold — replace with-group-by with a per-partition run-term invocation.
  2. Per-row iteration at scale — invoke a helper once per row in a multi-row solution table where K ≥ ~30 rows.

Outside these two shapes — for standard one-solution-in / one-solution-out helpers — use call. run-term against a captured in-memory helper in any other context produces a 500 or engine spin-out / ECONNRESET.

with-limit / with-skip / with-sort

with-limit limits the number of solutions returned. with-skip skips the first N results. Both thread their parameters into CouchDB view queries and only work when the runtime resolves the term through a view index — see the Query Execution doc for the full strategy table. with-sort controls sort direction; "desc" sets descending: true on the CouchDB view request.

;; Pagination: skip 20, take 10
(with-skip 20
  (with-limit 10
    (Qo.Db.Prop/prop-vals FolderId PageId PropName PropVal)))

;; Descending with limit
(with-sort "desc"
  (with-limit 5
    (Qo.Db.Prop/prop-vals FolderId PageId PropVal)))
